iSCSI boot for OpenSolaris Preliminary Users’ Guide

iSCSI boot for OpenSolaris Preliminary Users’ Guide

Pending official documentation, this page is offered as a guide to Caiman users who want to try OpenSolaris on an iSCSI boot target.

OpenSolaris and the Automated Installer

In this release, the Automated Installer is required to install OpenSolaris on an iSCSI boot disk target using the Automated Installer (AI).  AI reads the iSCSI parameters from the AI manifest (or optionally, through DHCP), mounts the iSCSI target disk using the OpenSolaris iSCSI initiator, and installs OpenSolaris on the disk.  Upon reboot after installation, the initiator in the iSCSI firmware mounts the target boot disk and OpenSolaris is booted.

Prototype status

iSCSI boot is released initially as a prototype.  Known problems are listed below in the "Caveats" section.

iSCSI boot parameters:

There are 4 parameters for specifying the location of the iSCSI target.  target IP, target name are mandatory.  Port number defaults to 3260, and LUN to 0.  LUN is sometimes necessary to distinguish between multiple boot disks on the same target.

The iSCSI boot parameters must be present:

  • to AI during installation, and
  • in firmware during the actual iSCSI boot of OpenSolaris

The iSCSI parameters are normally provided to AI through the manifest.  The only current exception to this is that under x86, AI can fetch them from DHCP Rootpath of the PXE boot interface when supplying the AI manifest <target_device_iscsi_parameter_source> value set to "DHCP".

Target configuration:

For OpenSolaris and Solaris, use iscsitadm(1m) to create the iSCSI boot targets.  Record the information used in iscsitadm to fill out the iSCSI parameters in the AI manifest.  Once the target is created, use "iscsitadm list target -v" to fetch the system-assigned iSCSI target name.  The iSCSI IP address will be the IP address of the system on which you created the target.  If you specified a LUN other than 0 (zero) in the target (usually to differentiate between virtual disks on the same target), that value will be entered as the LUN in the AI manifest.  The port should presently be only the default of 3260 and need not be provided.

Target configuration using OpenSolaris commands zfs and iscsitadm:

Using OpenSolaris commands, there are three types of targets you can create.

  • Create iSCSI target with ZFS volume:
        zfs create -o shareiscsi=on -V 13G rpool/iscsi
  • Create iSCSI target at mountpoint:
        iscsitadm create target -z 12G -b /mntiscsi localname
  • Create iSCSI target on raw device:
        iscsitadm create target --type raw -b /dev/dsk/cNtNdNpN localname

For an example of what a configured target looks like (here, a ZFS volume):
 $ pfexec iscsitadm list target -v
 Target: iscsipool/ipool
     iSCSI Name: iqn.1986-03.com.sun:02:c9e1533d-2d61-c198-b4ea-a9e3c40a49dd
     Alias: iscsipool/ipool
     Connections: 1
         Initiator:
             iSCSI Name: iqn.1986-03.com.sun:01:00e0815b0dd6.4b1fd234
             Alias: unknown
     ACL list:
     TPGT list:
     LUN information:
        LUN: 0
             GUID: 600144f04afab9de0000e0815cee3b00
             VID: SUN
             PID: SOLARIS
             Type: disk
             Size:   20G
             Backing store: /dev/zvol/rdsk/iscsipool/ipool
             Status: online
The "iSCSI Name" from the output line 2 is used as the iSCSI target name (NOTE: not the initiator iSCSI name). The IP address of the target is also used in the AI manifest.  LUN is taken - here it is the default of zero.  The port is optional as well, since the default is 3260.  Here is what the ai_target_device section of the AI manifest looks like:

<ai_target_device>
    <target_device_iscsi_target_name>
       iqn.1986-03.com.sun:02:c9e1533d-2d61-c198-b4ea-a9e3c40a49dd
    </target_device_iscsi_target_name>
    <target_device_iscsi_target_ip>10.0.0.1</target_device_iscsi_target_ip>
    <target_device_iscsi_target_port>3260</target_device_iscsi_target_port>
    <target_device_iscsi_target_lun>0</target_device_iscsi_target_lun>
</ai_target_device>

For non-OpenSolaris and Solaris targets, you will need to provide the same iSCSI parameters in the AI manifest. To determine the iSCSI target name, check the documentation for the software that you used.

Required packages in AI manifest:

Two packages are required for iSCSI boot:  SUNWiscsi (initiator and administration) and SUNWiscsidm (data mover).  They should be included in any iSCSI boot manifest along with other ai_install_packages:

<ai_install_packages>
  ...
  <pkg name="SUNWiscsi/>
  <pkg name="SUNWiscsidm/>
</ai_install_packages>

The packages are in the default manifest, default.xml.  The user must also provide them in any custom manifests for OpenSolaris to reboot from an iSCSI target.

Network parameter configuration in firmware:

x86:

On X86, the PC BIOS firmware on a NIC (network interface card) must be manually programmed with the initiator and target parameters, or be configured to receive them from DHCP.  It may also require non-standard firmware to be loaded.  To support iSCSI boot, the NIC must support iBFT, iSCSI Boot Firmware Table.  The Intel® PRO/1000, driver name e1000g, was used for AI development.  For basic networking configuration, you can configure the initiator and target separately - statically or through DHCP.  The initiator network configuration will be fetched and used by OpenSolaris - no further OpenSolaris networking configuration is needed (except for special cases described below).  See also the links section.

SPARC:

OBP firmware supporting iSCSI boot must be installed.  Niagara 2 and later SPARC hardware is supported.  The networking parameters for initiator and target must be set either:

  • by typing them on the "boot net" command line
  • set the network-boot-arguments OBP variable before typing "boot net" (either from OBP with setenv network-boot-arguments <args> or from OpenSolaris with "eeprom network-boot-arguments="<args>", where <args> is a comma-separated list of OBP parameters
  • on the DHCP server (see caveat below)

Both the initiator and target parameters are set through either DHCP or through OBP - they cannot be a combination of DHCP and OBP.

NOTE: OBP firmware supporting iSCSI boot is not yet released.

OpenSolaris Networking configuration:

NOTE: Currently, AI officially supports only networking configuration through NWAM and DHCP.  Because of compatibility problems between NWAM and iSCSI boot in general, as well as a panic in SPARC iSCSI boot over DHCP, manual configuration might be preferable.  This is addressed with the caveats and in the static configuration section below.

Presently, the Automated Installer is limited to installing network configurations with:

  • NWAM, the Network Auto-Magic service (as opposed to network/physical:default or no network configuration service)
  • DHCP (as opposed to static configuration)

However, since firmware establishes the TCP connection that iSCSI boot uses, the network interface itself requires no other configuration and is actually best left unaltered by OpenSolaris and NWAM.  For this reason, users may choose to avoid NWAM until NWAM is more aware of iSCSI boot, which is planned for NWAM phase 1.

OpenSolaris DHCP server configuration for AI

installadm commands do part of the DHCP administrative work for AI, but an additional DHCP macro symbol, Rootpath, can provide iSCSI boot parameters automatically so they don't necessarily have to be defined in firmware.

DHCP configuration with AI's installadm(1m)

NOTE:  iSCSI boot is not integrated into installadm, which configures DHCP, so separate DHCP server configuration is necessary for custom manifest configuration of iSCSI boot parameters. 

To insure that your iSCSI boot target disk is used by one and only one initiator, you can associate the iSCSI boot parameters with a specific AI manifest.  To have the DHCP server provide the iSCSI boot parameters for a specific MAC address, you can first use installadm to create a DHCP macro for that MAC address, then modify the DHCP configuration for that MAC address using dhtadm or dhcpmgr to specify the iSCSI target parameters.

installadm create-client method

installadm create-client -e <macaddr> -n <svcname> -t <image_path>
 This will make a DHCP macro named after the MAC address prefixed with "01" (zero-one).  After installation, the DHCP symbol Rootpath can be set to provide the iSCSI boot parameters in this format:
    iscsi:<target IP>::<port>:<LUN>:<target name>
 Port and LUN are optional and will default if left blank.
 The -M option of dhtadm can be used to add (and remove) DHCP symbols.
 For example, to add the iSCSI boot definition to the Rootpath symbol for an existing macro for a MAC address:

  dhtadm -M -m 01001517C7B56C -g \ 
    -e Rootpath="iscsi:10.0.0.1::::iqn.1986-03.com.sun:02:c9e1533d-2d61-c198-b4ea-a9e3c40a49dd"

For dhcpmgr, edit the properties of the macro named after the MAC address, adding Option of "Rootpath" with Value formatted the same as above.
 "Rootpath" must begin with the characters "iscsi:" for the iSCSI parameters to be recognized.

AI custom manifest method:

If you would prefer to add a custom manifest for a MAC addresss instead of creating a custom service, you can use "installadm add":
 installadm add -m <manifest> -n <svcname>
 In the example below, a MAC address is specified so that the manifest will only apply to the system with that LAN card.  This prevents other systems from using the same iSCSI target.

  <ai_criteria_manifest>
      <ai_criteria name="mac">
          <value>0:e0:81:5b:d:d6</value>
      </ai_criteria>
      <ai_target_device>
          <target_device_iscsi_target_name>
               iqn.1986-03.com.sun:02:c9e1533d-2d61-c198-b4ea-a9e3c40a49dd
          </target_device_iscsi_target_name>
          <target_device_iscsi_target_ip>10.0.0.1</target_device_iscsi_target_ip>
      </ai_target_device>
...

If you use this method, you must manually create a separate macro for the MAC.  The general command for adding a macro for a particular MAC address is:
     dhtadm -A -m <MAC> -g \
        -d :Include=<MACRO>: Rootpath=\"iscsi:<target IP>::<port>:<LUN>:<target name>\":
The dhcpmgr Java GUI can also be used.
Use the "installadm add" method only if you are more experienced with AI manifests and DHCP.

DHCP for x86:

PXE booting and booting from the iSCSI boot NIC are console-selectable at boot time.  Instructions differ depending on PC BIOS, platform, NIC, etc.

Manual networking configuration

AI does not yet support static networking configuration; however, a few factors favor manual configuration at this time:

  • SPARC DHCP panics in tests with current firmware
  • NWAM, which AI relies on to perform the networking configuration (usually with DHCP), can take the link down, having the effect of disconnecting the boot disk completely since OpenSolaris reads and uses the networking parameters from the firmware table, iBFT, static networking configuration for OpenSolaris is not necessary
  • two DHCP addresses will be consumed by a single computer

OpenSolaris provides two services for networking configuration: nwam and network/physical:default.  The nwam service invokes the nwam daemon, nwamd(1M). 

Until static configuration options are given in AI:

  • NWAM will have to be disabled manually, and
  • DNS must be configured manually

Advantage in using DHCP to provide the iSCSI boot parameters:

  • the user does not have to provide them.  The sysadmin can configure the AI server and DHCP server according to the MAC address of the NIC.

Disabling NWAM and enabling network/physical:default manually

Services can be enabled and disabled using the service repository for the OpenSolaris just installed.  First, disable automatic reboot by setting "ai_auto_reboot" to "false" in the AI manifest in the "ai_manifest" section.  After AI completes successfully, and before the initial reboot, manually mount the new BE, edit the service configuration, and dismount the BE before rebooting:

  beadm mount opensolaris /a
  svccfg
  repository /a/etc/svc/repository.db
  select nwam
  setprop general/enabled=false
  select network/physical:default
  setprop general/enabled=true
  exit
  
  beadm umount opensolaris
  reboot

NWAM must be disabled before reboot, but DNS can either be configured before or after reboot.  DNS files nsswitch.conf must be modified and resolv.conf must be created.  If done after reboot, the DNS client might need to be restarted manually.

For network/physical:default, the IP address is passed on from firmware to OpenSolaris, so no additional configuration is needed, whether the IP configuration is static or uses DHCP.

Bug to watch:  6914190 Support for unmanaged network interfaces in NWAM - hands off

Setting DNS after reboot:

An easy way to configure the DNS client service:

  • Put name server info into /etc/resolv.conf:
       nameserver <DNS server IP #1>
       nameserver <DNS server IP #2>
       domain <domain name>
  • cp /etc/nsswitch.dns /etc/nsswitch.conf

Then the DNS client service might have to be started, since it won't run without resolv.conf (state:offline):
svcadm enable dns/client

The initiator settings will be adopted by OpenSolaris when it is iSCSI booted.  You can verify with ifconfig and "netstat -r" after reboot.  This "magic" is done by strplumb.

SPARC OBP static configuration might look like this:
 ok setenv network-boot-arguments iscsi-target-ip=10.0.0.1,iscsi-target-name=iqn.1986-03.com.sun:01:00e0815b0dd6.4b1fd234,host-ip=10.0.0.2,subnet-mask=255.255.255.0,router-ip=10.0.0.1
ok boot net

You can also use eeprom(1M) to set OBP network-boot-arguments.

Troubleshooting:

During troubleshooting, you may need to use the OpenSolaris initiator, iscsiadm(1m), to look at what AI has mounted, or perhaps to test mounting manually.  "iscsiadm add static-target <target name>,<target IP>[:port]".  "iscsiadm list target -S" lists the OpenSolaris device name of a mounted disk.

snoop can help you monitor iSCSI traffic.  "snoop -d <IF> -o /dev/null tcp port 3260" is the basic command to see packets using the default iSCSI port 3260.  If you direct output to a file, wireshark(1) can be used to analyze it graphically.

Be certain that packages SUNWiscsi and SUNWiscsidm are both present in the ai_package_list section in your manifest.

To monitor the DHCP server, stop the DHCP server service with svcadm disable -t dhcp-server and run it in a console window manually with /usr/lib/inet/in.dhcpd -d

Caveats:

Currently, AI uses PXE boot and DHCP with NWAM for networking configuration.  Several caveats must be mentioned here.

PXE boot vs. iSCSI boot.

x86:  Most, and perhaps all currently supported iSCSI NICs have separate firmware for PXE and iSCSI boot.  AI requires PXE boot, so there must be separate NIC cards to perform these functions during AI.

AI is currently dependent on DHCP and NWAM, but NWAM is inherently unreliable for iSCSI boot because traffic can be rerouted when the datalink used for iSCSI goes down.  After NWAM takes the iSCSI boot interface down, the system will run, but the system disk will be inaccessible.  NWAM Phase 0.5 allows only one link to be active at at time.  This issue will be addressed in Phase 1 of NWAM.

SPARC DHCP panics since it can't mount the zfs root pool:

6910958 DHCP iSCSI boot on OpenSolaris panics: can not read pool label from /iscsi-hba/disk

The main side effect of this bug is that DHCP("boot net:dhcp") cannot be used for either IP address assignment or iSCSI boot parameters on SPARC; static OBP firmware configuration must be used as described above in the "boot net" command.

Bug 3136- TI fails to create VTOC on unlabeled disk on Sparc

An unlabeled SPARC disk will still fail on AI - the disk must be manually labeled.

Another SPARC DHCP issue, currently unresolved, is that the same command to launch AI will launch iSCSI boot: "boot net:dhcp".  If the DHCP Rootpath="iscsi:...", the iSCSI definition will take precedence and AI will not start, so AI has to be installed before DHCP Rootpath is defined and the system rebooted into OpenSolaris.  This can be inconvenient and hazardous if the user has an installation overwritten because Rootpath was not yet set up by the sysadmin.

6910970 iSCSI-booted OpenSolaris freezes after power management idle
 Workaround:  in /etc/power.conf - add or edit: "autopm disable", then run pfexec pmconfig.

Packages SUNWiscsi and SUNWiscsidm must be provided in any AI manifest that is used to iSCSI boot.

Until the NWAM and iSCSI technologies converge, using the Gnome desktop to configure the network will be problematic.  Switching between manual and automatic configuration will result in OpenSolaris being hung, requiring a hard reboot.  Any networking changes involving NWAM or the iSCSI boot interface must be done from live media or from the console after AI has completed and before initial reboot.  Refer to the section on manual network configuration above.

Network Interface Cards

x86: Installer support was developed using an Intel® Pro/1000 PT server adapter NIC with iSCSI firmware manually loaded.  Chipset revision is critical: Intel® Server Adapters - Which adapters support Intel® iSCSI Remote Boot?  (see "Controller").  Some Broadcom NICs were reported to work with Solaris but have not been tested under OpenSolaris.

For SPARC, it is a matter of what OBP version supports iSCSI boot as well as machine class. Only Niagara 2 and later sun4v boxes are now supported.  Presently, a prototype OBP is still required: 4.31 or later.  You will almost certainly have to flash the OBP along with other system firmware on your platform.

NOTE:  the official release of iSCSI boot in OBP is scheduled for version 4.32, which has not yet been released.

Logical Domains (LDOMS):

Early testing shows that an iSCSI target boots from an LDOM.

The following configuration worked on a Huron 5220 and LDOMS version 1.2:
  ldm add-vds primary-vds0 primary
  ldm add-vcc port-range=5000-5100 primary-vcc0 primary
  ldm add-vsw net-dev=e1000g1 primary-vsw0 primary
  ldm set-mau 1 primary
  ldm set-vcpu 1 primary
  ldm set-memory 2G primary
  ldm add-config initial

  shutdown -y -g0 -i6
  ifconfig vsw0 plumb
  ifconfig e1000g1 down unplumb
  ifconfig vsw0 192.168.100.251 netmask 255.255.255.0 broadcast + up
  cat >/etc/hostname.e1000g1 <<EOF
  > 192.168.100.251
  > EOF
  svcadm enable vntsd
  ldm add-domain ldg1
  ldm add-vcpu 1 ldg1
  ldm add-memory 2G ldg1
  ldm add-vnet vnet1 primary-vsw0 ldg1
  ldm set-var auto-boot\?=false ldg1
  ldm bind-domain ldg1
  ldm list-domain ldg1

  telnet localhost 5000
  ldm start-domain ldg1
  boot net:dhcp (do iSCSI AI)

VirtualBox

VirtualBox iSCSI initiator:

The VirtualBox iSCSI Initiator can be used to install to an iSCSI target, but presently there is no way to iSCSI boot through VirtualBox.

NOTE:  for more on the booting issue, see VirtualBox bug 5935.
