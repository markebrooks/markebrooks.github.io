OpenSolaris のビルドとインストール (パート1)

OpenSolaris のビルドとインストール (パート 1)

著者:Rich Teer
2005 年 6 月

はじめに

これは、OpenSolaris のソースコードを入手してビルドする方法について解説した 2 つの記事のうちの最初のものです。最初の記事では、必要な背景情報 (専門用語、ツールの入手場所など) を提供し、基本的なコンパイルとインストールについて説明します。2 番目の記事では、より複雑なコンパイルとインストールについて説明します。これらの記事の内容は執筆時点では正確なものでしたが、OpenSolaris は進化し続けています。最新の詳細情報については、ON 開発者参照情報を参照してください。

これらの記事は、OpenSolaris のビルドとインストールの方法について解説したものであって、OpenSolaris 開発者向けのガイドとして使用するものではありません。したがって、OpenSolaris のビルドとインストールに必要とされる以上の情報は収録されていません。ただし、ほかのほとんどの大規模なオープンソースプロジェクトと同様に、この情報を必要とするアクティブな OpenSolaris 開発者の数は、自己啓発のためにこれをビルドする人たちの数に比べておそらく少ないため、このことを問題視する必要はありません。

これらの記事は、大規模なソフトウェアプロジェクトのビルドに関する基本的な知識と、C プログラミングについての多少の知識を前提としています。「Hello World」のコンパイルに奮闘している人が OpenSolaris をコンパイルしようとするとは思えません。ただし、Solaris のビルドに関する知識はないことを想定しており、必要な手順はすべて解説します。

専門用語

OpenSolaris のビルド方法について説明する前に、いくつかの用語を定義する必要があります。このリストをできるだけ簡潔にしておくため、OpenSolaris のビルドとインストールに直接関連している用語だけを記述します。ここで解説する用語は、OpenSolaris の開発者が使用する用語の一部です。必要に応じて、新しい用語を紹介するときにそれらをときどき定義します。

定義する必要のある最初の用語は、OpenSolaris そのものです。OpenSolaris の Web サイト jp.opensolaris.org/os/about/ には、Sun が承認した公式の定義が収録されていますが、この記事では、「OpenSolaris」とは Sun がリリースしてきた Solaris のソースコードと、そのソースコードからビルドされたバイナリを指します。したがって、OpenSolaris のソースコードのコピーを持っている、または OpenSolaris を実行している (後者は前者からビルドされる) と言うことができます。OpenSolaris の立ち上げにあたり、Sun は OpenSolaris の独自の「ディストリビューション」をリリースするつもりはありませんが、そのことでほかのユーザーが独自のディストリビューションをリリースできなくなるわけではありません (著者は OpenSolaris を使ってビルドされるディストリビューションプロジェクトが最低 3 つはあり、それらに PowerPC プラットフォームへの移植までもが含まれていることを知っています)。

正確には、Solaris DVD または CD で出荷されるすべてのものが OpenSolaris であるわけではありません。この記事の執筆時点 (2005 年 6 月) では、OpenSolaris は ON コンソリデーションのみで構成されています。「コンソリデーション」 とは、一緒に開発および配布される、関連したソフトウェアコンポーネントのセットです。ON 以外に、コンソリデーションの例としては、ウィンドウシステム、Java Desktop System などがあります。「ON」は「OS/Net」または「OS/Networking」の略語であり、カーネル、ファイルシステム、基本コマンド、いくつかのモジュールとドライバ、デーモン、ヘッダーファイル、およびライブラリを含むコンソリデーションです。ちなみに、Sun が出荷する Solaris DVD および CD の内容はひとまとめにして「WOS (Wad Of Stuff)」として知られています。これは、Solaris 製品を構成するすべてのコンソリデーションを合わせたものです。

これまでのところ、ON を構成するすべてのコンポーネントがソース形式で利用できるわけではありません。その理由については、jp.opensolaris.org/os/no_source/ に説明されています。時間の経過とともに、ほとんど (すべてではないが) のクローズドソースはオープンソースコード (Sun が開発したコードまたは OpenSolaris コミュニティーが寄与したコード) に置き換えられますが、そうなるまでは、ビルド済みのバイナリオブジェクトが、OpenSolaris のビルドを容易にするために提供されます。

OpenSolaris は開発の最先端であるため、新たにコンパイルされたバージョンをインストールすると、マシンの「ブリック化」が発生する可能性があります。つまり、マシンがブート不能になったり、使用できなくなったりします。不適切なインストールや壊れたコンポーネントのインストールを行うと、マシンが「ウォームブリック」になることがあります。この状態から回復するには、通常、ブートできることがわかっているほかのメディアからブートするのが適切な方法です。

OpenSolaris を扱うときに目にすることがある最後の 1 組の用語は、Nevada と Tonic です。「Nevada」は、Solaris 10 のあとに続く Solaris リリースの内部名です。OpenSolaris は Solaris Nevada ソースに基づいています (つまり、OpenSolaris のソースコードは Solaris 10 の First Customer Shipment (FCS) バージョンに基づいているのではなく、Solaris 10 が進化したものなのです)。「Tonic」は、OpenSolaris プロジェクトの内部名です。したがって、この用語を使用するのは、おそらく Sun の従業員とパイロットプログラムの参加者だけです。

OpenSolaris を使って作業する際に認識しておく必要のあるいくつかの専門用語についての説明が終わったので、実際のビルドに必要な前提条件を見ていきましょう。

前提条件

OpenSolaris のビルドに最初に必要なものは、適切なマシンです。一般的には、Solaris 10 を実行できるコンピュータであれば十分です。つまり、SPARC のビルドには、200 MHz 以上の UltraSPARC-II またはそれより新しいプロセッサを搭載したマシンが必要です。x86 にはさまざまなハードウェアがあるため、ビルドの要件は少々複雑になります。この記事で x86 とある場合、特に明記しないかぎり、32 ビットの x86 プラットフォームと 64 ビットの AMD64 プラットフォームの両方を意味します。AMD64 プラットフォームはマーケティングタイプによって x64 と呼ばれることもあります。しかし、Solaris x86 HCL (ハードウェア互換リスト) に載っていれば、それなりに最近のハードウェアで問題ありません。HCL は、www.sun.com/bigadmin/hcl/ にあります。

現在では、SPARC と x86 の両方のアーキテクチャーがサポートされていますが、クロスコンパイルはサポートされていません (つまり、OpenSolaris を配備するプラットフォームはそのビルドに使用するプラットフォームと同じでなければなりません)。OpenSolaris ソースツリーのビルドはかなり長時間にわたるプロセスなので、十分な仕様のマシンを使用することを推奨します (著者は、x86 のビルドには 2G バイトの RAM と 7200 RPM ハードドライブが搭載された Acer Ferrari 3400 を使用し、SPARC のビルドには 2 つの 450 MHz プロセッサと 2G バイトの RAM が搭載された Ultra 60 を使用しました)。この記事を執筆した時点では、使用するマシンは Solaris Express: Community Release (SXCR) Build 16 以降を実行している必要があります。ただし、SXCR の最新のビルドを「強く」お勧めします。より新しい OpenSolaris のバージョンでは最小要件としてより新しい SXCR のバージョンが要求される可能性が高いからです。

Solaris、OpenSolaris、および Solaris Express という用語は初心者の方の混乱を招く可能性があるため、これらの内容について 簡単に説明します。まず第一に、Solaris とは Sun から提供される商標の付いた製品です。新規バージョンは 2 ～ 3 年ごとに出荷され、アップデートはおよそ四半期ごとに発行されます。本稼働環境で使用するべきなのは、このバージョンの OS です。

Solaris Express は、現在の開発ツリーのスナップショットで、約 1 か月ごとに発行されます。リリース日と以前の Solaris Express との間で「最良の」ビルド (もっとも安定し、問題の少ないもの) がリリースされるビルドとなります。Solaris Express のコードには十分なテストを行なっていますが、ベータ品質である (Sun の通常の厳しいテスト手順を経ていない) ため、本稼働で使用してはならないことを記憶に留めておくことが重要です。とはいえ、Solaris Express は Solaris の新機能や改善された機能を利用できるもっとも簡単で、おそらく最良の方法です。たとえば、Solaris Express のユーザーは、FCS バージョンが利用できるようになる約 2 年前から、Solaris 10 に導入された機能 (ゾーンや DTrace のような機能を含む) を使用していました。著者の経験では、Solaris Express は非常に安定しています。Solaris Express のビルドは、Sun の最新エンジニアリングビルドを数バージョン (1 ～ 2 か月) のみの遅れで追従します。例として、この記事の執筆時点では、Solaris Express は Nevada のビルド 15 に基づいていましたが、この時点のエンジニアリングバージョンはビルド 16 でした。前述したように、Solaris Express には Community Release もあり、これはビルドごとにリリースされます。SXCR は標準の Solaris Express に比べて受けるテストが少なくなっています。また、OpenSolaris の開発者だけを対象としているので、相応の注意を払って配備する必要があります。

前述のように、OpenSolaris は Sun エンジニアリングからの現在の最先端のビルドに基づいています。まさにその性質により、OpenSolaris は Solaris Express に比べて受けるテストが少ないため、扱いと配備に若干の困難が伴います。重要な本稼働に使用するものでは決してありません。

最後に、十分なディスク容量が必要です。ソースコード、クローズドバイナリ、およびツールには約 90M バイト必要です。また、各ビルドツリーには最低でも 1.5G バイト必要です。このことを念頭に置いて、ビルドする予定のツリーごとに約 2G バイトの空きディスク容量を用意する必要があります。おそらく、OpenSolaris に組み込まれるソースコードがさらに増えれば、必要なディスク容量も増えるでしょう。

Solaris のバイナリディストリビューションのインストール

前述したように、OpenSolaris をビルドするためのマシンは、適切な最新バージョンの Solaris Express または SXCR (あるいは、以前の OpenSolaris ビルド) を実行している必要があります。著者は Developer クラスタのみがインストールされているマシンで OpenSolaris のビルドに成功しましたが、Sun では全クラスタがインストールされていることを推奨しています。

Solaris のインストール方法については、記事の範囲外となります。Sun の『Release and Installation Collection』 (docs.sun.com) に詳しく解説されています。

前提条件 (必要な場合は Solaris のバイナリディストリビューションのインストールを含む) を満たしたら、必要な各種コンポーネントおよびツールのダウンロードをいつでも開始できます。

必要なコンポーネントのダウンロード

OpenSolaris のコンポーネントやツールをダウンロードする前に、作業用のディレクトリを作成するようにしてください。このディレクトリとその子ディレクトリは「ワークスペース」と呼ばれます。次の例では、ホームディレクトリにある open_solaris というディレクトリをベースワークスペースとして使用します。このディレクトリには、ビルドごとにサブディレクトリを作成し、そこに格納するビルドにちなんで名前を付けます (例: build-17)。各 build ディレクトリには、testws というビルド用のワークスペースを作成します。

これにより、次のようなディレクトリ階層が作られます。

$HOME/open_solaris ベースディレクトリ

$HOME/open_solaris/build-17 ビルド 17 のソースとその他のコンポーネント

$HOME/open_solaris/build-n ビルド n のソースとその他のコンポーネント

$HOME/open_solaris/build-n/testws コンパイル用のワークスペース

OpenSolaris の Web サイト (おそらくほかのサイトでも OpenSolaris コンポーネントを提供していますが、この記事では jp.opensolaris.org を使用するものとします) から「build」ディレクトリにダウンロードする必要があるファイルは多数あります。最低でも、ソースコード、クローズドソースコードからビルドされたバイナリ、ON ビルドツール、コンパイラなどのツールは必要です。リリースノートもダウンロードするようにしてください。

次の例では、Nevada の ビルド 17 をコンパイルするものとします。また、Solaris Express: Community Release ビルド 16 (Nevada ビルド 16) はすでにインストールされているものとします。ビルド 17 を使用するため、ベースワークスペースは $HOME/open_solaris/build-17 となり、ダウンロードするファイルはこの場所に格納するようにしてください。簡潔にするために、次の例ではこのディレクトリを $OPEN_SOL と呼びます。

コンパイラとその他のツールのインストール

必要なファイルのダウンロードが完了したら、コンパイラと ON ビルドツールをインストールする必要があります。後者は、特に OpenSolaris のビルドとインストールに使用するプログラムとスクリプトで構成されます。コンパイラとその他の Sun Studio ツールは、tar ball またはパッケージセットのどちらかで提供され、ON ビルドツールはパッケージとして提供されます。次のすべての例では、ファイル名は実在のものを表していますが、実際のファイル名はここで示すファイル名とはおそらく異なっています。たとえば、ファイル名の多くは日付スタンプを取り入れていますが、これは場合によって変わります。

まず、パッケージ SUNWonbld をインストールして、ON ビルドツールを追加します。

# cd /tmp
# bzip2 -dc $OPEN_SOL/SUNWonbld-20050322.i386.tar.bz2 | tar xf -
# cd onbld
# pkgadd -d . SUNWonbld
# cd ..

簡潔にするために、pkgadd コマンドからの出力は省略しました。また、ON ビルドパッケージの実際のファイル名はおそらく異なっています。これは、各ビルドの要件を反映するように、これらのツールが頻繁に更新されるためです。

次に、Sun Studio コンパイラとその他のツールの tar インストールイメージをインストールします。

# cd /opt
# bzip2 -dc $OPEN_SOL/sunstudio10-20050613.x86.tar.bz2 | tar xf -

コンパイラとツールのインストールが完了したら、それらのディレクトリがユーザーの PATH に含まれていることを確認し、OpenSolaris の実際のビルドを行うためのディレクトリを作成する必要があります (この作業を root ではなく自身のユーザーとして行ったあと、$OPEN_SOL/build-n に戻ります)。

$ export PATH=/opt/onbld/bin:/opt/SUNWspro/bin:$PATH
$ mkdir testws

適切な PATH が含まれるようにログインスクリプトを変更するのも一案です。

ソースコードとクローズドバイナリの展開

次のステップでは、クローズドバイナリとソースコードを展開します。

$ cd testws

$ bzip2 -dc ../opensolaris-closed-bins-20050322.i386.tar.bz2 | tar xf -

$ bzip2 -dc ../opensolaris-src-20050322.tar.bz2 | tar xf -

OpenSolaris コードのビルド

OpenSolaris ソースコードのビルドの準備が整うまで、あともう少しです。ビルド環境の準備にはスクリプトを使用します。このスクリプトは多数の環境変数を設定します。このスクリプトをユーザーの環境に合わせてカスタマイズする必要があります。このスクリプトのローカルコピーを作成してから、それを変更します。

$ cp usr/src/tools/env/opensolaris.sh .

$ chmod 644 opensolaris.sh

opensolaris.sh を編集します。必要に応じて、環境変数 GATE、CODEMGR_WS、STAFFER、および VERSION の定義を変更してください。STAFFER、CODEMGR_WS、および VERSION 以外の環境変数はデフォルト値のままとします。STAFFER、CODEMGR_WS、および VERSION は、それぞれ「rich」、「/home/rich/open_solaris/build-17/$GATE」、および「OpenSolaris_build-17」に変更します (言うまでもありませんが、CODEMGR_WS の値は適宜変更するようにしてください)。

最後に、nightly スクリプトを使用して、実際のビルドを行います (nightly は、カーネルだけでなく、すべてのコンポーネントをビルドします)。

$ nightly opensolaris.sh

このステップにはしばらく時間がかかることがあるため、ほかに行うべきことを見つけるのに良い時間です (著者の 2 GHz Ferrari 3400 ラップトップでは、このステップが完了するのに約 90 分かかります)。ビルドが完了すると、3 つのログファイルがディレクトリ log/log.mmdd (ここでの mm は月、dd は日にち) に生成されます。これらのログファイルは、mail_msg、nightly.log、および proto_list_i386 (または類似したもの) です。最初のファイルには、完了時に $STAFFER に送信される電子メールのコピーが含まれており、ビルドの概要が記載されています。2 番目のファイルにはビルドプロセスの詳細ログが含まれており、最後のファイルはパッケージの作成に使用できるプロトタイプリストです (詳細については、この記事では触れない)。

mail_msg ファイルにエラーが含まれていない場合は、ビルドが正常に行われたことになります。ビルドが正常に行われなかった場合は、nightly.log を調べて、その原因を突き止めることができます。

OpenSolaris のインストール (パート 1)

すべてがうまくいったと想定して、新たにコンパイルしたコンポーネントをインストールできます。新しいコンポーネント (特に、カーネルまたは重要なライブラリやアプリケーション) のインストールはブリック化を引き起こすことがあるため、Install (cap eye install と呼ばれる) ユーティリティーを使用します。このユーティリティーは、新しくビルドしたカーネルの tar ball を作成します。この tar ball を既存のものと並行してインストールできます (ビルドしたマシン以外のマシン上でテストすることを強く推奨します)。Install では新しいユーザーランドのコンポーネントをインストールしないので注意してください。新しいカーネルとモジュールだけがインストールされます。Install プロシージャーでは、現在インストール済みのカーネルが上書きされることがないため、ブリック化の危険が緩和されます。

root になり、使用しているワークスペースディレクトリ (前に $OPEN_SOL と呼ばれていたもの) に移動します。/opt/onbld/bin および /opt/SUNWspro/bin に存在するツールとコンパイラが root の PATH に含まれていることを確認します (または単にコマンドのフルパス名を使用します)。

最初に、bldenv スクリプトを実行して、必要なすべての環境変数が適切な値に設定されていることを確認する必要があります。

# bldenv opensolaris.sh

次のステップでは、Install tar ball を作成し、ビルドと同じマシンでテストしていることを前提として、それをインストールします。

# Install -k i86pc -G kernel-17

# cd /

# tar xf /tmp/Install.root/Install.i86pc.tar

Install コマンドの -k フラグはインストールしているカーネルアーキテクチャーを通知し、-G フラグは glomname を指定します。後者は、新しいカーネルが格納される /platform/<arch> のサブディレクトリ名であり、この場合は /platform/i86pc/kernel-17 (SPARC では、現在 /platform/sun4u/kernel-17) に格納されます。glomname を指定しない場合は、新しくコンパイルしたカーネルによって現在のカーネルが上書きされます。これによってブリック化が発生した場合は、代替メディアからのブートによって回復するしか方法はありません。

x86 では、新しいカーネルをブートする前に実行しなければならない作業がもう 1 つあります。tar ball が抽出されたら、新しいカーネルディレクトリの名前を /boot/solaris/filelist.ramdisk ファイルに追加する必要があります。SPARC プラットフォームでインストールを行っている場合、このステップは該当しません。SPARC プラットフォームでは GRUB ブートローダーを使用しないためです。

あとは、新しいカーネルを使ってブートするだけです。

# reboot -- kernel-17/amd64/unix

この例では、64 ビットの amd64 マシンを使用しているものとします。64 ビットの SPARC マシンでは kernel-17/sparcv9/unix、32 ビットの x86 マシンでは kernel-17/unix をそれぞれ使用します。

すべてうまくいった場合は、opensolaris.sh の VERSION 変数に入力した文字列が起動バナーに表示されるはずです。

rich@orac3559# uname -a

SunOS orac 5.11 OpenSolaris_build-17 i86pc i386 i86pc

一部のマシンでは、OpenSolaris の一環として提供されないデバイスドライバが欠落しているために、この方法でブートできない場合があります。この場合は、元となる OS のインストールで提供されるモジュールとドライバをユーザーのテストディレクトリにコピーし、それをインストールするという方法を試してみてください。

この対処方法を次に示します。

# cd /platform/i86pc

# rm -fr kernel-17/*

# (cd kernel; tar cf - .) | (cd kernel-17; tar xf -)

# cd $OPEN_SOL

# Install -k i86pc -G kernel-17

# cd /

# tar xf /tmp/Install.root/Install.i86pc.tar

# reboot -- kernel-17/amd64/unix

新しいカーネルだけでなく、そのほかのコンポーネントもインストールする場合は、BFU と呼ばれるプロセスを使用する必要があります。BFU については、もう一つの記事で説明します。BFU はよく注意して使用するようにしてください。 最新の最先端のコンポーネントがどうしても必要な開発者だけにお勧めします。

まとめ

この記事では、OpenSolaris を扱うときに目にする専門用語の一部を定義しました。また、OpenSolaris のソースコードとそのコンパイルに必要なその他のコンポーネントの入手方法について説明し、最後に、OpenSolaris の簡単なコンパイルとインストールに必要な手順について説明しました。これらの手順は、次のとおりです。

  1. ソースコード、クローズドバイナリ、コンパイラ、およびビルドツールを jp.opensolaris.org またはそのいずれかのミラーから入手します。
  2. ON ビルドツールと Sun Studio コンパイラを /opt にインストールし、それらのディレクトリをユーザーの PATH に追加します。
  3. ソースコードとクローズドバイナリの tar ball を展開します。
  4. opensolaris.sh スクリプトをコピーして編集します。
  5. nightly スクリプトを使用して、ソースコードをビルドします。
  6. Install スクリプトを使用して、結果として生成された新しいカーネルをインストールします。
  7. 新しいカーネルを使ってリブートします。

推薦ドキュメント

Web サイト opensolaris.org には、『OpenSolaris 開発者向けリファレンス・ガイド』などのいくつかのドキュメントが含まれています。また、OpenSolaris Tools Community サイト、特に Sun Studio ダウンロードサイト (jp.opensolaris.org/os/community/tools/sun_studio_tools/) についても詳しく確認するようにしてください。著者の書籍『Solaris Systems Programming』 [Teer 著、2005 年] は、ユーザーランドの OpenSolaris コードを使って作業する読者には不可欠です。また、Jim Mauro 氏および Richard McDougall 氏による 『SOLARIS インターナル - カーネル構造のすべて』 [Mauro、McDougall 共著、2001 年] は、OpenSolaris のすべてのカーネルハッカーの書棚に置くことをお勧めします。コミュニティーのブログ (jp.opensolaris.org/os/blogs/) は、もう一つの重要な情報源です。

[Mauro、McDougall 共著、2001 年]: 『SOLARIS インターナル - カーネル構造のすべて』 ISBN 4-89471-458-2。詳細は、www.solarisinternals.com/ を参照してください。

[Teer 著、2005 年]: 『Solaris Systems Programming』 ISBN 0-201-75039-2。詳細は、 www.rite-group.com/rich/ssp/ を参照してください。
