2 前提条件

この章では、OpenSolaris を開発、ビルド、およびテストするためのハードウェアとソフトウェアの要件について説明します。このような要件の多くは、どのような Solaris または OpenSolaris のインストールでも簡単に満たすことができます。興味の対象によって変わりますが、自分では利用できないハードウェアまたはソフトウェアで作業をテストする必要があるかもしれません。このような場合はプロジェクト・リーダーに相談してください。OpenSolaris コミュニティの開発者は、企業または組織 (Sun を含む) が提供するさまざまな設備を利用できます。自分のプロジェクトに使用できる資源について知りたいプロジェクト・リーダーの方は、community underline liaison at opensolaris dot org まで問い合わせてください。

2.1 ハードウェア

このセクションでは、OpenSolaris のハードウェアの要件について詳細に説明します。プロジェクトによっては要件が追加されることもあります。たとえば、ドライバの開発には、当然、当該デバイスが必要です。したがって、ここに記載された情報は、決定的な規則としてではなく、ガイドとして使用してください。たとえば、パッチなどの作業をリモートの環境に転送すればビルドまたはテストが可能なので、OpenSolaris が動作しないシステムでも開発できます。便宜上、このセクションでは、Solaris または OpenSolaris が動作するシステムで開発とビルドを行うことを想定します。

2.1.1 開発環境

開発環境に適用される最も簡単な要件は、ソース・ツリーを格納できるだけの十分な領域があること、そして、テキスト・エディタや cscope などのツールを実行できるだけの十分な CPU とメモリがあることです。現在のソース・ツリーは約 300M バイトです (ソース・コード管理メタデータを除く)。したがって、格納するツリーごとに少なくとも 300M バイトの空間を割り当てる必要があります (フルセットの SCCS メタデータを含む場合は、約 1.9G バイトが必要)。同じシステムでツリーをビルドする場合は、さらに空間を追加する必要があります。詳細は、セクション 2.1.2 を参照してください。

一般的に、Solaris 10 または OpenSolaris を実行するコンピュータはすべて、この目的に適しています。このドキュメントを作成している時点では、UltraSPARC II 以降の CPU (つまり、200MHz より高速な CPU すべて) を搭載するすべての SPARC システムがサポートされます。Solaris 10 がサポートされる SPARC プラットフォームのリストについては、[[http://docs.sun.com/source/817-6337-06/install-solaris.html>>#http___docs_sun_com_source_817633706_installsolaris_html]] を参照してください。i386 と amd64 のシステムの場合、ハードウェア・サポートは少々複雑になります。x86 ハードウェアの互換性についての詳細は、[[http://www.sun.com/bigadmin/hcl/>>#http___www_sun_com_bigadmin_hcl_]] を参照してください。現在または将来のハードウェア・サポートについては、Solaris の最新のリリース・ノートを必ず調べてください。最新のリリース・ノートは、[[http://docs.sun.com/>>#http___docs_sun_com_]] にあります。

OpenSolaris がサポートするハードウェアは、最新の Solaris リリースまたはアップデートとは異なる場合があります。OpenSolaris のハードウェア・サポートについての最新情報は、利用可能になり次第、[[http://opensolaris.org/>>#http___opensolaris_org_]] に登録されます。

表: 開発システムの要件


 ~----------------------------------------------------------------
 CPU            サポートされる任意の CPU
 ~----------------------------------------------------------------
 メモリ         128MB を推奨
 ~----------------------------------------------------------------
 スワップ       要件なし
 ~----------------------------------------------------------------
 ストレージ     ソース・ツリーのコピーごとに 300MB から 1.9GB (*)
 ~----------------------------------------------------------------

(*) 合計のサイズは、使用するソース管理のタイプによって変わります。ソース管理データがない場合、ビルド・ツリーには約 3.5G バイトが必要です。CVS と SCCS にはさらに約 50M バイト以上が必要です。正確なサイズはリビジョン履歴の量によって変わります。

2.1.2 ビルド環境

OpenSolaris のように大規模で複雑なソフトウェアの一部のビルドは、メモリ、CPU、およびストレージを消費するプロセスです。ほとんどどのようなコンピュータでも OpenSolaris をビルドできますが、小さな構成では、非常に長い時間がかかってしまいます。dmake を使用して高度な並行ビルド処理を実行する場合は、特に、十分なスワップ空間が必要です。スワップ空間が十分でないと、ビルドは失敗します。次に、OpenSolaris をビルドするシステムのハードウェア要件を示します。

表: ビルド・システムの要件


 ~----------------------------------------------------------------
 CPU            300 MHz UltraSPARC または x86 CPU
                AMD Opteron または UltraSPARC III を推奨
 ~----------------------------------------------------------------
 メモリ         256MB が最小。1GB を推奨 (+)
 ~----------------------------------------------------------------
 スワップ       1GB が最小。2GB を推奨 (+)
 ~----------------------------------------------------------------
 ストレージ     ソース・ツリーのコピーごとに 3.5 から 5GB (*) (**)
 ~----------------------------------------------------------------

上記の最小ビルド・システムでは、フル・ビルドに約 14 時間かかります。増分ビルドにすると、少し短くなります。経験則で言うと、フル・ビルドにかかる時間の 1/3 です。

次の表に、代表的なシステムでフル・ビルドにかかる時間を示します。これは 2005 年 1 月 24 日時点の結果です。より多くのコードが追加され、クローズド・コンポーネントの数が減ると、ビルド時間は増えます。もちろん、ビルドにかかる実際の時間はシステムによって変わります。


 システム       CPU                       メモリ        時間
 ~----------------------------------------------------------------
 Ultra 2        300MHz UltraSPARC II x2   512MB        5時間9分
 ~----------------------------------------------------------------
 汎用           AMD    Athlon64 3200        1GB        1時間23分
 ~----------------------------------------------------------------

ビルド・システムにはマルチ CPU が効果的です。また、複数のマシンでコンパイル作業を分散することもできます。並行分散ビルドを構成する方法についての詳細は、「4 OpenSolaris をビルドする」を参照してください。

(*) 合計のサイズは、使用するソース管理のタイプによって変わります。ソース管理データがない場合、ビルド・ツリーには約 3.5G バイトが必要です。CVS と SCCS にはさらに約 50M バイト以上が必要です。正確なサイズはリビジョン履歴の量によって変わります。

(+) dmake(1) を並行モードで使用して、5 つ以上のジョブを同じマシンで実行する場合、より多くのメモリとスワップ空間が必要です。

() コンパイラとツールは /opt にインストールされます。x86 の場合は 300M バイト、SPARC の場合は 800M バイトがさらに必要です。

2.1.3 テスト環境

変更が発生した場合、その変更をテストする環境の要件はさまざまに変わります。アーキテクチャ固有のコードにまったく影響しない変更では、各アーキテクチャでサポートされる任意のシステムでテスト・スイートをビルドして実行するだけでテストできます。たとえば、新しい STREAMS モジュールをネットワーク・スタックに追加した場合、1 台の x86 システム (32 ビットと 64 ビットのカーネルとユーザ・プログラムをテストする) と 1 台の SPARC システムでビルドしてテストするだけでおそらく十分です 。その一方で、たとえば、x86 ブート・コードを変更した場合、より多くのハードウェアの組み合わせでテストして、機能の一部 (特に、目立たない部分) が壊れていないことを確認する必要があります。

次に、テストする上で考慮する必要があるハードウェア要因を示します。


        - アーキテクチャ: i386、amd64、SPARC。
        - メモリ・サイズ: その変更は 32MB で機能するか。32GB 必要か。
        - マルチプロセッサか、ユニプロセッサか。
        - グラフィック・コンソールか、シリアル・コンソールか、システム LOM コンソールか。
        - ディスクレス・システムか、「通常」のシステムか。

必ずしも、これらすべての要因がユーザの変更に該当するわけではなりません。したがって、できるだけ慎重に、変更が引き起こす影響について考える必要があります。一般的に、VM サブシステムやファイルシステム・サポートなどのコアとなるカーネル・コードを変更した場合は、考えられる最大限のハードウェアの組み合わせでテストする必要があります。その一方で、アドオン・ドライバやマシン固有のコードを変更した場合は、(ほとんどの場合) 関連するハードウェアをテストするだけでかまいません。詳細は、第 5 章を参照してください。

入手できるハードウェアでは完全にテストできない変更を行う場合が多々あります。このような場合は、[[http://opensolaris.org/>>#http___opensolaris_org_]] 経由で連絡してください。OpenSolaris コミュニティには、変更をテストするためのハードウェアを提供可能な企業または組織 (Sun を含む) が参加しています。

2.2 オペレーティング環境パッケージ

この章の残りでは、OpenSolaris をビルドしてパッケージ化するのに必要なソフトウェアについて説明します。ソースを表示して編集し、作成したパッチをほかの人に送信するだけの場合、このセクションを読まなくてもかまいません。ソースを表示したり編集したりするだけならば、特別なパッケージは必要ありません。ただし、開発向けに使用しようとするシステムに Solaris または OpenSolaris を初めてインストールする場合は、Entire クラスタ (SUNWCall) をインストールすることをお勧めします。これは、あとで OpenSolaris をビルドすることを決めた場合に、時間を大幅に節約できるためです。

まず、Entire パッケージ・クラスタ (SUNWCall) は、開発、ビルド、およびテスト向けのすべてのシステムにインストールすることを強くお勧めします。Developer (SUNWCdev) パッケージ・クラスタをインストールしたシステムでもかまいませんが、この構成はテストされておらず、評価も推奨も行われません。次に、SUNWonbld パッケージを入手してインストールする必要があります。このパッケージの usr/src/tools には、OpenSolaris をビルドするときに必要なツールのビルド前のバージョンが入っています。代わりに、nightly(1) に -t オプションを付けて実行し、これらのツールをビルドしてもかまいません (「4.2 nightly と bldenv を使用する」を参照)。ただし、依存関係の問題を避けるためにも、上記パッケージをインストールすることをお勧めします。SUNWonbld は [[http://opensolaris.org/os/downloads/>>#http___opensolaris_org_os_downloads_]] からダウンロードできます。最後に、Sun Studio コンパイラや dmake などのツールを入手してインストールする必要があります。これらのツールは、[[http://opensolaris.org/os/community/tools/sun_studio_tools/>>#http___opensolaris_org_os_community_tools_sun_studio_tools_]] から入手できます。

テスト参加者のための注意: (始める前に) SUNWonbld とコンパイラは 1 つの標準 tarball に結合されています。インストールするパッケージはありません。

たとえば、Solaris または OpenSolaris の CD または DVD をフル・インストールして使用する場合、プログラム、ライブラリ、およびカーネル・コンポーネントは完全に同じものです。ただし、あとで BFU (「5.3 BFU を使用して ON をインストールする」を参照) または Install (「5.2 Cap-Eye-Install を使用してカーネルをインストールする」を参照) を使用して、OpenSolaris の一部をアップグレードする場合、あるいは、自分独自で更新したファイルを少しだけビルドしてコピーする場合、まず、より新しいバージョンのシステム・パッケージを 1 つまたは複数インストールする必要がある場合があります。このような場合を「Flag Day」と呼びます。Flag Day についての詳細は、「5.1.3 Flag Day とその他の危険要因」を参照してください。

2.3 コンパイラを入手してインストールする

必要なコンパイラは、OpenSolaris をビルドするプラットフォームによって変わります。OpenSolaris の x86 ビルドは現在、デフォルトで、Sun One Studio と GNU コンパイラの両方を使用します。これは、32 ビット・オブジェクトをビルドするのに cc と CC を使用し、64 ビット・オブジェクトをビルドするのに gcc を使用するためです。これは、ビルド 19 で変更される予定です。gcc の動作を制御するには、環境変数 GNUC と GNUC64 を設定します。詳細は、「4.2.3 変数」を参照してください。

OpenSolaris をビルドするのに使用するコンパイラには特別なパッチが必要になる可能性があるため、OpenSolaris の Web サイトでは常に、Studio コンパイラの現在のバイナリへのリンクを管理します。ビルド 73 以降の Solaris には、デフォルトの x86 ビルト用に必要な GNU C コンパイラとそれに関連するユーティリティが付属しています (場所は /usr/sfw) 。ただし、このコンパイラは SPARC や i386 のオブジェクトをビルドするには十分でなく、また、ビルドするには OpenSolaris のソースをカスタマイズする必要もあります。ほかのバージョンのコンパイラ、アセンブラ、リンカー、make(1)、または dmake(1) の使用はサポートされていません。このようなバージョンを使用すると、ビルドに失敗したり、正しく機能しないバイナリを生成したりする可能性があります。したがって、ビルドに成功するためには、必ず、[[http://opensolaris.org/os/downloads>>#http___opensolaris_org_os_downloads]] にあるコンパイラを使用する必要があります。ときどき必須ツールが更新されることがありますが、このような場合、各ユーザに通知が行われ、適宜、新しいバージョンが利用できるようになります。

このドキュメントの作成時点では、主要な OpenSolaris のソースをビルドするのに使用できるコンパイラは、[[http://opensolaris.org/os/community/tools/sun_studio_tools/>>#http___opensolaris_org_os_community_tools_sun_studio_tools_]] にある Sun Studio 10 ツールだけです。GNU Compiler Collection (gcc) でビルドしてもかまいませんが、そのためには、追加のツールとソースの修正が必要になります。詳細は、[[http://opensolaris.org/os/community/tools/gcc/>>#http___opensolaris_org_os_community_tools_gcc_]] を参照してください。使用するコンパイラにかかわらず、ON 固有ビルド・ツール (onbld) のセットが必要になります。これらのツールは、SPARC と x86 の両方のプラットフォーム用とも、[[http://opensolaris.org/os/downloads/>>#http___opensolaris_org_os_downloads_]] にあります。

Studio 10 コンパイラをインストールする方法は 2 つあります。インストール・イメージからインストールする方法と、完全製品としてインストールする方法です。インストール・イメージとは、Studio 10 製品の内容のほとんど (すべてではない) を入れて圧縮したシンプルな tar ファイルのことで、すでにパッチや有効なライセンス・キーが適用されています。この内容は、ON をビルドするのに十分です。内容についての詳細は、[[http://opensolaris.org/os/community/tools/sun_studio_tools/faqs/>>#http___opensolaris_org_os_community_tools_sun_studio_tools_faqs_]] を参照してください。インストール・イメージを抽出すると、実行したディレクトリの下にある opt/SUNWspro というディレクトリに抽出されます。上記のようにルート・ディレクトリで抽出した場合 (推奨)、完全製品と同じ場所にインストールされます。インストール・イメージにはパッチを追加して適用する方法がないので注意してください。

例: Studio 10 コンパイラをインストールする (インストール・イメージの場合)

ここでは、インストール・イメージを /var/tmp/sunstudio10-sparc.tar.bz2 としてダウンロードし、/opt/SUNWspro (推奨) にインストールすると仮定します。このインストールを実行するには、次のように入力します。


 $ su
 Password:
 # cd /
 # bzip2 -dc /var/tmp/sunstudio10-sparc.tar.bz2 | tar xf -

完全な製品には、独自のインストーラとインストールの説明が書かれている完全なマニュアルが付属しています。インストールの説明の詳細については (たとえば、グラフィカルなインストール方法やコマンドによるインストール方法)、[[http://docs.sun.com/app/docs/doc/819-0485>>#http___docs_sun_com_app_docs_doc_8190485]] を参照してください。完全な製品を使用する場合は、OpenSolaris をビルドする前に、必要なパッチをインストールする必要があります。通常、このようなパッチはダウンロードした製品に含まれています。必要なパッチについての詳細は、[[https://opensolaris.org/os/community/tools/sun_studio_tools/>>#https___opensolaris_org_os_community_tools_sun_studio_tools_]] を参照してください。デフォルトでは、Studio 10 製品は /opt/SUNWspro にインストールされます。Sun Workshop/Forte/Studio がすでにインストールされており、設定を変えたくない場合を除いて、この場所は変更しないでください。

例: Studio 10 コンパイラをインストールする (完全な製品の場合)

ここでは、完全な製品を /var/tmp/sunstudio10-x86.tar としてダウンロードし、/var/tmp/studio-install に展開し、また、「ABC123」というライセンス・キーを受け取っていると仮定します。このインストールを実行するには、次のように入力します。


 $ mkdir /var/tmp/studioinstall
 $ cd /var/tmp/studioinstall
 $ tar xf /var/tmp/sunstudio10-x86.tar
 $ su
 Password:
 # ./batch_installer -s ABC123
 ...
 # cd patches
 # patchadd -M . patchlist
 ...

2.4 カスタム・ツールを入手してインストールする

一般的な開発ツールとコンパイラに加えて、onbld と呼ばれるカスタム・ツールのセットが必要です。このセットには、コンパイラとアセンブラの互換性、CTF データ、ABI 監査、ON bits (コード) のインストールなどを処理するスクリプトとプログラムが含まれます。これらのユーティリティの多くはプラットフォーム固有であるため、onbld のパッケージ (SUNWonbld) は、サポートされるプラットフォームごとに提供されます (現在のところ、x86 と SPARC)。これらのツールは、ON ソース・ゲートの usr/src/tools サブディレクトリにあるソースからビルドされていることに注意してください。ソースの場所についての詳細は、3.2 の「ワークスペースの概要」を参照してください。ソースを利用できるのであれば、SUNWonbld をインストールしなくてもかまいません。しかし、ON インストールを最初から行うときに、依存関係の問題が発生する可能性があります。このパッケージは、[[http://opensolaris.org/os/downloads>>#http___opensolaris_org_os_downloads]] から入手できます。ある範囲の ON ソースをビルドするには、あるバージョンの onbld が必要になることがあることを覚えておいてください。つまり、古いバージョンの OpenSolaris で作業する必要がある場合は、そのバージョンに適合する onbld のバージョンを使用していることを確認してください。

ON 固有のビルド・ツールは、サポートされるビルド・マシン・アーキテクチャごとに、SVR4 (pkgadd(1M)) パッケージとして提供されます。このパッケージは /opt/onbld にインストールされます。

例: ON ビルド・ツールをインストールする

ここでは、ビルド・ツール・パッケージを /var/tmp/SUNWonbld-20050613.tar.bz2 としてダウンロードし、それをインストールしようとしていると仮定します。このインストールを実行するには、次のように入力します。


 $ cd /var/tmp
 $ bzip2 -dc SUNWonbld-20050613.i386.tar.bz2 | tar xf -
 $ su
 Password:
 # pkgadd -d . SUNWonbld

2.5 環境変数

ソース・コード管理およびビルド・ツールを正しく使用するためには、いくつかの環境変数を正しく設定する必要があります。これらの環境変数のほとんどは、ワークスペースに固有 (3.3.1 を参照)、またはビルドに固有 (4.1 を参照) のものです。ただし、インストールされたツールを正しく使用するためには、PATH 環境変数を正しく設定する必要があります。これは通常、ログイン・スクリプトの一部として自動的に設定されます。

最新の推奨される PATH 構成要素については、ゲートの public/README ファイルを参照してください。このファイルのコピーは、[[http://opensolaris.org/os/community/on/gate_info/README>>#http___opensolaris_org_os_community_onnv_gate_info_readme]] にあります。

PATH は、シェルや他のプログラムによって使用され、プログラムがどこにあるかを調べます。この機能は、OpenSolaris とはまったく関係ありません。しかし、ビルド・ツールに必要なさまざまなプログラムを見つけるためには、通常のシステム・ディレクトリに加えて、いくつかのディレクトリを PATH に指定する必要があります。次に、一般的な PATH の例を示します。


        /opt/SUNWspro/bin
        /opt/onbld/bin
        /opt/onbld/`uname -p`/bin

コンパイラまたは onbld ツールをデフォルト以外の場所にインストールした場合 (非推奨)、それに従って上記場所を変更する必要があります。また、PATH には必ず /usr/ccs/bin を指定します。さらに具体的には、/usr/ucb というディレクトリが PATH に指定されている場合は、/usr/ccs/bin を /usr/ucb の前に指定する必要があります。次に、これらのディレクトリがすべて指定されている PATH の例を示します。


        $ echo $PATH
        /usr/bin:/usr/sbin:/usr/ccs/bin:/usr/dt/bin:/usr/openwin/bin: \
            /opt/onbld/bin:/opt/onbld/bin/sparc:/opt/sfw/bin:/usr/sfw/bin

詳細は、ゲートの public/README ファイルを参照してください。

/usr/ucb

 を PATH に指定することは推奨されません。また、ビルド・プロセスでは、そのディレクトリにあるどのようなプログラムも必要ありません。何らかの理由で、そのディレクトリを含める必要がある場合は、PATH の最後に指定する必要があります。そうしなければ、/usr/ucb を PATH に指定した場合、特に、/usr/sbin の前に指定した場合、ビルドは失敗します。

いくつかのツールへのパスは /usr/src/Makefile.master やその他の makefile に明示的に指定されているため、PATH 環境変数を変更しても、意図どおりにならないことがあります。特に、コンパイラのパスは固定されています。コンパイラのパスを変更する必要がある場合は、make マクロを変更し、コンパイラの場所を指定する必要があります。詳細については、セクション 4.1 を参照してください。

PATH に加えて、/opt/onbld/man と /opt/SUNWspro/man を MANPATH に加えると便利です。こうしておけば、この文書で説明されている多くのユーティリティに関連するマニュアルページを簡単に表示できます。また、man(1) を呼び出すときに、-M /opt/onbld/man または -M /opt/SUNWspro/man を指定しても、これらのマニュアルページを表示できます。
