OpenSolaris のビルドとインストール (パート2)

OpenSolaris のビルドとインストール (パート 2)

著者:Rich Teer
はじめに
これは、OpenSolaris のビルドとインストールの方法について解説した 2 つの記事のうちの 2 番目のものです。最初の記事では、必要な背景情報 (専門用語、ツールの入手場所など) を提供し、基本的なコンパイルとインストールについて説明しています。最初の記事で実行した手順だけではニーズが十分に満たされていない場合があります。この記事では、このような不足している点に対処するために必要な作業について説明します。
この記事では、読者が最初の記事の内容をよく理解していること、つまり少なくとも OpenSolaris のソースコードの入手方法、ビルド方法、およびインストール方法を認識していることを前提としています。最初の記事の掲載から次の記事の掲載までの間に新しいバージョンが使用できるようになったとしても、一貫性を保つために、この記事では最初の記事で扱ったものと同じバージョンのコードを使用します。
これらの記事の内容は執筆時点では正確なものでしたが、OpenSolaris は進化し続けています。最新の詳細情報については、ON 開発者参照情報を参照してください。
要点のまとめ
最初の記事では、OpenSolaris を扱うときに目にする専門用語のほとんどを定義しました。また、OpenSolaris のソースコードとそのコンパイルに必要なその他のコンポーネントの入手方法について説明し、最後に、OpenSolaris の簡単なコンパイルとインストールに必要な手順について説明しました。これらの手順は、次のとおりです。

  1. ソースコード、クローズドバイナリ、コンパイラ、およびビルドツールを jp.opensolaris.org またはそのいずれかのミラーから入手します。
  2. ON ビルドツールと Sun Studio コンパイラを /opt にインストールし、それらのディレクトリをユーザーの PATH に追加します。
  3. ソースコードとクローズドバイナリの tar ball を展開します。
  4. opensolaris.sh スクリプトをコピーして編集します。
  5. nightly スクリプトを使用して、ソースコードをビルドします。
  6. Install スクリプトを使用して、結果として生成された新しいカーネルをインストールします。
  7. 新しいカーネルを使ってリブートします。

新しいカーネルをビルドしてテストするだけなら、この手順でかまわないのですが、それでは不十分な場合があります。新しいユーザーランドのコンポーネントをいくつかインストールする場合や、新しくビルドしたカーネルが、現在インストールされているユーザーランドのライブラリやアプリケーションと互換性がない場合です。後者のシナリオは、Flag Day と呼ばれます。Flag Day のもう一つの例は、ライブラリと、そのライブラリに依拠しているコマンドを同時に更新する必要がある場合です。
Flag Day
「Flag Day」とは、ON コンポーネントの新しいバージョンを使用する前に、新しいバージョンのシステムパッケージを 1 つ以上インストールしておく必要がある場合です。この状況が発生すると、Flag Day のお知らせが OpenSolaris の Web サイト (jp.opensolaris.org/os/community/onnv) に掲載されます。これらのお知らせには、通常の Solaris パッケージツールと互換性がない可能性のある、必要な新規ソフトウェアの構築やインストールのための手順が記載されています。特に、コンポーネントを更新するための BFU ユーティリティー (この記事で取り上げられている) を使用すると、それらのコンポーネントは標準の Solaris パッケージでアップグレードできなくなります。これが心配の原因となる場合は、最新の Solaris Express リリースを使用することがおそらく最良の方法です。実のところ、BFU の使用は、アクティブな ON コード開発者 (または常に最先端を走り続けなければならない方) にしかお勧めしていません。
BFU の概要
すべての新しい ON コンポーネントをインストールするためのプロセスを「BFU」と呼びます。BFU は、Blindingly Fast Upgrade または Bonwick-Faulkner Upgrade (当初の開発者 Jeff Bonwick 氏と Roger Faulkner 氏にちなんでつけた名称) の略です。BFU は、1 組の cpio アーカイブを使って、システムの既存のコンポーネントを直接上書きします (これが、一度 BFU を実行すると、通常の Solaris パッケージツールが使用できなくなる理由です)。これらのアーカイブは、nightly ビルドプロセスの一環として作成されます。このプロセスについては、前の記事に説明してあります。
新たにコンパイルしたコンポーネントを、BFU を使ってインストールする前に、3 つの環境変数を定義しておく必要があります。これらの環境変数は次のとおりです。
FASTFS 通常は /opt/onbld/bin/`uname -p`/fastfs に設定します。
BFULD 通常は /opt/onbld/bin/`uname -p`/bfuld に設定します。
GZIPBIN 通常は /bin/gzip に設定します。
特に BFU を頻繁に計画している場合は、これらの環境変数をシェルの初期設定ドットファイル (.profile など) 内に設定しておくことが、おそらくもっとも便利な方法です。前の記事で説明した opensolaris.sh 環境ファイルでも、これらの変数を設定できます。bfu を実行する前に、bldenv opensolaris.sh を (まず usr/src/tools/env/opensolaris.sh からコピーしてから、必要に応じて編集して) 使用するのがおそらく最良の方法です。新しいソースデリバリのたびに opensolaris.sh のコピーを更新する習慣をつけておくのも一案です。こうしておけば、環境変数が新しく追加された場合に、自然と気付くことができるからです。
BFU を使用して ON をインストールする
BFU を使用して OpenSolaris の ON 部分をインストールするのは、比較的簡単な作業です。1 つの引数 (インストールするアーカイブのパス) を指定して bfu コマンドを呼び出すだけです。
前の記事では、$HOME/open_solaris/build-17/testws をワークスペースディレクトリとして使用しました。同じディレクトリを引き続き使用していると想定して、次のように bfu を実行できます。
# bfu `pwd`/archives/`uname -p`/nightly
システムファイルが上書きされるので、bfu を root として実行してください。また、/opt/onbld/bin が root の PATH に含まれていることを確認してください。

BFU プロシージャー (著者の Acer Ferrari 3400 ラップトップでは約 2 分かかる) が完了すると、新しく作成されたサブシェルに切り替わります。これは、新しいコマンドやライブラリが現在実行中の古いカーネルと互換性がない可能性があるからです。新しいサブシェルでは、その PATH が /tmp/bfubin に設定され、LD_LIBRARY_PATH が /tmp/bfulib に設定されています。これらのディレクトリには、問題を解決してシステムをリブートするために通常必要とされる旧バージョンのコマンドとライブラリが入っています (/net/greenline.eng へのアクセスに関するエラーは、bfu.sh に含まれるバグ 4865419 によるもので、無視してかまいません)。BFU を実行したあとは、システムのリブートに init または shutdown を使用しないようにしてください。代わりに、reboot コマンドを使用してください。

ブリック化が起きないようにする場合は、システムをリブートする前に、これらの競合を解決しておく必要があります。
競合を解決する
Solaris を実行しているどのマシンにも多数の構成ファイルがあり、そのほとんどはデフォルトのインストールから変更されます。これらの構成ファイルの例として、/etc/auto_home、/etc/nsswitch.conf、および /etc/path_to_inst があげられます。bfu コマンドは、「競合データベース」と呼ばれる、これらのファイルのリストを保持します。
BFU が構成ファイルを上書きすると、現在のファイルのコピーがまず /bfu.child の下に保存されます。また、cpio アーカイブからの新しいファイルのコピーが /bfu.parent の下に置かれます。以前の bfu の /bfu.parent (存在する場合) は /bfu.ancestor (本質的には親の親) に移されます。これらのディレクトリの下にあるファイルは、それぞれ「子」、「親」、「祖先」と呼ばれます。
cpio アーカイブの抽出が終わると、bfu はこれらのファイルのさまざまなバージョンの差分処理を行い、次の規則に従います。

  • ファイルが変更されていない (つまり、親と子が同じ) 場合は、実行することはないため、レポートされるものもありません。
  • ファイルが、前回の BFU の実行時に親から承認された (つまり、子が祖先と同じ) 場合は、現時点で親が自動的に承認され、ファイルは「update」と指定されます。  これは、BFU をはじめて実行したときにも起こることであり、ほかの規則は適用されません。
  • ファイルが前回のファイルの先頭と同じ場合、BFU は、ユーザーがファイルの最後に行を追加した (つまり、子が親と連結行で構成される) とみなします。子のバージョンが復元され、ファイルは「restore」と指定されます。 
  • 親と子は異なっているが、親と祖先が同じである場合は、競合は過去のものとみなされ、ファイルは「old」と指定されます。 
  • 最後に、親と子が異なっており、親と祖先も異なっている場合は、競合は新しいものとみなされ、ファイルは「NEW」と指定されます。 

したがって、「NEW」の競合は、デフォルトとは異なるファイルを指すため、デフォルトは更新されたことになります。これを解決するには、新規ビルドで導入された変更内容を既存のファイルに移行する必要があります。
詳しい調査を行わないで親を承認することも可能ですが、その場合、子に対して行われたカスタマイズがすべて失われます。これらのカスタマイズの多くは、ON 以外のパッケージからクラスアクションスクリプトによって実行されるため、親をやみくもに承認すると、多くの時間が無駄になることがあります。したがって、競合は手動または自動のどちらかで解決するようにしてください。この方法についてはあとで説明しますが、すべての競合が自動的に解決できるとは限らない (あるいは、それらを手動で解決することにした) 場合、適切なベースライン競合データベースを持つことはとても良い考えです。そして、これを実現するには、システムで現在実行されているリリースに対応する BFU アーカイブをインストールします。
別の手段ですでにインストールしてある同じビルドに対して BFU を実行するときは、現在の構成ファイルが正常に動作することがわかっているため、すべての競合を無視してかまいません。それ以降のビルドに対して BFU を実行する場合は、その前にリブートする必要があります。
自動の競合解決
ほとんど (すべてではないが) の競合は、acr と呼ばれるスクリプトを使って自動的に解決できます。このスクリプトは通常、BFU 実行後の保護環境で呼び出されます。この方法で呼び出された場合は、コマンド行オプションを指定する必要はありません (特殊な用途の場合、acr は 1 つまたは 2 つのコマンド行パラメータを使用します。最初のパラメータには代替ルートディレクトリの名前を指定し、2 番目のパラメータには BFU アーカイブが含まれているディレクトリの名前を指定します)。
acr は、conflict_resolution.gz というファイルを使用します。このファイルは、-a オプションを指定して nightly を実行するたびに作成され、BFU アーカイブが含まれているディレクトリに格納されます。acr スクリプトは、処理しているファイルのリストを標準出力に表示し、詳細な結果を allresults というファイルに書き込みます。このファイルは、/tmp のサブディレクトリに格納されます。次に、acr 呼び出しの例を示します。

手動の競合解決
手動で競合を解決するには、問題となっているファイルの祖先と親のバージョンの差分処理を行います。より一般的には、次のコマンドを実行し、差分処理を手動で適用します。
bfu# diff /bfu.ancestor/$file /bfu.parent/$file
BFU の実行が今回、はじめての場合は、/bfu.ancestor が存在しないため、前述のように、ベースライン競合データベースを作成しておくことをお勧めします。多くのファイルに対しては、ショートカットを使用できます。変更内容の大半が追加や変更であり、BFU プロセスによって /bfu.conflicts というディレクトリに切り替わる場合、次のコマンドを実行し、
bfu# pw
/bfu.conflicts
bfu# diff $file /$file
差分がすべて右向きの矢印 (「<」ではなく「>」) になっていることを確認すれば、うまく行きます。次に例を示します。

この例では、重要な差分の行が右向きの矢印になっているので問題ありません。
ほとんどの場合はこのプロセスが有効ですが、例外がいくつかあります。次に、その例外について詳しく説明します。これらの例外は、etc/name_to_major、etc/security/*_attr、および /etc/path_to_inst です。
etc/name_to_major ファイルは、デバイス名をそのメジャー番号に対応づけます。各デバイスには、一意のメジャー番号が割り当てられている必要があります。まずは、祖先と親とを差分処理で比較すると良いでしょう。デバイス名は重要ですが、メジャー番号はそれぞれ異なります。サポートされなくなったデバイスは削除することをお勧めしますが、残しておいても、通常は問題ありません。確かでない場合は、残しておいてください。
新しいデバイスは、未使用のメジャー番号 (可能であれば、差分処理から得られる番号) を割り当てて追加するようにしてください。
終了したら、次の 2 つのソートコマンドを実行して、変更内容を確認できます。
bfu# sort -k1,1 /etc/name_to_major | sort -uc -k1,1
bfu# sort -k2n /etc/name_to_major | sort -uc -k2n
最初のコマンドは最初に重複したデバイス名をレポートし、2 番目のコマンドは重複したメジャー番号のうちでもっとも低いものをレポートします。リブートの前に、どちらのコマンドからも出力がない状態にすべきです。リブートの際にカーネルがこのような競合について警告しますが、そのときにはすでに遅すぎることがあります。
etc/security/*_attr ファイルは、クラスアクションスクリプトによって大幅に変更されるため、親と子のバージョンも大きく異なることがあります。このことは、前述の差分処理のショートカットにあまり効果がないことを意味します。ただし、親と祖先に関するより一般的な差分処理は引き続き有効です。
最後の例外は、/etc/path_to_inst です。実行している内容を正確に把握していないかぎり、このファイルには触れないようにしてください。
一般に、「NEW」と指定されている競合だけを調べる必要があります。
BFU とゾーン
BFU を実行しているマシンが大域ゾーンのほかにゾーンをホストしている場合は、もう少し作業を行う必要があります。ゾーンを作成するたびに大域ゾーンの内容がコピーされるため (大域ゾーンが「空白 (sparse)」でない場合)、すべての非大域ゾーンの内容が大域ゾーンと整合性が取れていることを確認する必要があります。幸運にも、大域ゾーンが更新されたあとで各ゾーンを更新することにより、BFU が代わりにこの処理を行います。各ゾーンには大域ゾーンとは異なる構成ファイルが含まれていることがあるため、各ゾーンには独自の BFU 競合管理ディレクトリセットがあります。BFU を実行する前に、すべてのゾーンをシャットダウンすることが得策です。これにより、新しいファイルによって依存関係が侵害されても、ゾーンが破壊されずに済むからです。また、BFU を実行したあとに新しいゾーンを追加する場合は、bfu を再実行して、新しいゾーンが確実に更新されるようにすることも一案です。
単独のゾーンを対象に BFU を実行することはできません。ゾーン内のユーザーランドへの変更をテストする場合は、ビルドワークスペースからそのゾーンにファイルを手動でコピーする必要があります。
まず大域ゾーンで競合を解決し、その後それぞれの非大域ゾーンで競合を解決してから、リブートする必要があります。ゾーン内の競合の解決は、大域ゾーンと同じ方法で行われますが、多くのファイルが大域ゾーンの対応するファイルと同じであるため、手動で結合する必要のあるファイルはほとんどありません。大域ゾーンからファイルをコピーすれば十分です。
大域ゾーンとすべての非大域ゾーン内の競合がすべて解決したら、システムをリブートできます。非大域ゾーンを再起動する前に、システムをリブートしてください。
ビルドを行わずに BFU を実行する
これまでは、OpenSolaris の副産物として自分自身でビルドしたアーカイブを使って BFU を実行することを想定していました。最新のコンポーネントをコンパイルしないで使用する場合は、どうなるでしょうか。幸いなことに、ビルド済みの BFU アーカイブを利用できます。ソースからビルドしないで BFU を実行するには、BFU の対象となるリリースに対応している BFU アーカイブと ON ビルドツールをダウンロードする必要があります。現在のビルドと BFU の対象となるビルドとの間にかなりの相違があるときは、場合により BFU を数回実行する必要があることも承知しておいてください。
最近 BFU を実行したビルド 17 のシステムに対して BFU を実行してビルド 18 にする例を見てみましょう。このシリーズのパート 1 と同様に、2 つの必要なファイルを $HOME/open_solaris/build-18 というディレクトリにダウンロードし、カレントディレクトリがそのディレクトリになっているものと想定します。便宜上、このディレクトリを $OPEN_SOL と呼びます。
まず、BFU アーカイブを展開します。
$ bzip2 -dc opensolaris-bfu-20050720.i386.tar.bz2 | tar xf -
これにより、archives-20050720 というディレクトリが作成され、この中に BFU アーカイブが格納されます (このファイルや類似したファイルの名前はリリースごとに変わります)。ここで、root になり、ON ビルドツールをインストールする必要があります。
# cd /tmp
# bzip2 -dc $OPEN_SOL/SUNWonbld-20050720.i386.tar.bz2 | tar xf -
# cd onbld
# pkgadd -d .SUNWonbld
簡潔にするため、pkgadd からの出力は省略しました。次に、前述の例に類似したコマンドを使って BFU を実行します (まず FASTFS、BFULD、および GZIPBIN が適切に定義されていることを確認し、次に /opt/onbld/bin がユーザーの PATH に含まれていることを確認します)。
# bfu $OPEN_SOL/archives-20050720/`uname -p`
次に、競合があれば解決し (この例では競合はありません)、リブートします。
bfu# acr
No conflicts to resolve.
bfu# reboot
updating /platform/i86pc/boot_archive...this may take a minute
/platform/i86pc/boot_archive の更新に関するメッセージは、x86 固有のものです。すべてが順調に進んでいると想定して、マシンは新しいビルド (この場合は、ビルド 18) でブートします。いつものように確認するには、ログインして、uname を実行します。
$ uname -a
SunOS orac 5.11 opensol-20050720 i86pc i386 i86pc
まとめ
この記事では、自分自身でビルドしたソース、あるいは Web サイトの opensolaris.org (またはほかの場所) からダウンロードした BFU アーカイブのどちらかから BFU を使用して OpenSolaris のバージョンを更新する方法について説明しました。前者の例として、前回の記事でビルドしたビルド 17 に対して BFU を実行しました。後者の例として、ビルド 17 から BFU を実行してビルド 18 にしました。
自分自身でビルドしたソースから BFU を実行するための手順は、次のとおりです。

  1. FASTFS、BFULD、および GZIPBIN 環境変数が正しく設定されていることを確認します。
  2. ワークスペースディレクトリから root として bfu コマンドを実行します。その際、コマンド行オプションとしてアーカイブの名前を渡します。たいてい、これは `pwd`/archives/`uname -p`/nightly となります。
  3. 競合があれば、acr スクリプトを使って解決します。
  4. 残っている競合があれば、手動で解決します。
  5. reboot コマンド (init または shutdown ではなく) を使用してリブートします。

ビルド済みの BFU アーカイブのインストールに必要な手順は似ており、次のとおりです。

  1. Web サイト opensolaris.org またはそのいずれかのミラーから、BFU の対象となるビルド向けの BFU アーカイブと ON ビルドツールをダウンロードします。
  2. BFU アーカイブを展開します。
  3. ON ビルドツールをインストールします。
  4. FASTFS、BFULD、および GZIPBIN 環境変数が正しく設定されていることを確認します。
  5. root として bfu コマンドを実行します。その際、手順 2 で展開した BFU アーカイブの名前を渡します。
  6. 競合があれば、acr スクリプトを使って解決します。
  7. 残っている競合があれば、手動で解決します。
  8. reboot コマンドを使用してリブートします。

また、一度 BFU を実行すると、システムを更新するための通常の方法 (パッチやアップグレードインストールなど) はそれ以降使用できなくなることも警告しました。新規インストールを行うか、もう一度 BFU を実行する必要があります。最後に、BFU がアクティブな OpenSolaris 開発者だけを対象にしていることを記述しました。一般のユーザーは Solaris Express を使用すればおそらく十分です。
