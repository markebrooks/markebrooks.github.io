OpenSolaris をインストールしてテストする

5 OpenSolaris をインストールしてテストする

この章では、OpenSolaris bits をインストールするための柔軟性に富んだ方法をいくつか紹介します。 OpenSolaris には、現時点では稼動システムに必要となるすべてのプログラムが含まれていないので、これらの手順を正常に実行するためには、フル・インストール (通常は Solaris または Solaris Express) が必要になります。

さらに、カーネルとコア・ユーザランド・コンポーネントに関するテスト手順の一部についても説明します。 これらのテストは、システムを可能な限り対象とするよう考慮され、追加的なテストをインフラに記述したり追加したりできるよう十分な柔軟性も持ち合わせていますが、テストの大半はプロジェクト固有のテスト・スイートによって行われます。 バグを修正したり、新機能を追加したりする場合は、コードの owner(s) に連絡し、既存のテストを入手することをお勧めします。また、修正中のバグを検出したり、新機能の機能性を検証したりする、新しいテストを提供してください。

5.1 インストールの概要

特定のファイルをプロト・エリアから稼働システムに手動でコピーする以外に、システムに bits をインストールする方法は主に 3 つあります。 どの方法を使用するかは、何を変更したかによって異なります。 カーネルのみを変更した場合は、セクション 5.1.1 の Install に関する説明を参照してください。 ON のカーネルとユーザランド・コンポーネントを変更し、変更内容を同時に適用する必要がある場合は、Install を使用する前にユーザランドの変更内容をハードコピーするか、BFU を使用します。BFU に関する詳細については、セクション 5.1.2 を参照してください。

一部のソースが欠落していても完全なビルドが得られるように、一連のクローズド・バイナリが提供されています。また、ビルド・システムは、これらのバイナリを使用できるよう修正されています。 Install または正常に動作する BFU アーカイブを構築するには、ビルド製品とともに closed-bins.<プラットフォーム>.tar.gz コンポーネントを使用する必要があります。 クローズド・バイナリの構築と使用に関する詳細情報については、「4 OpenSolaris をビルドする」および最新のリリース・ノートを参照してください。

これらのインストール方法は、ますます複雑となる上に多くの時間を必要としますが、システムの大部分をアップグレードすることができます。 大半の ON 開発者は ON ソースのみを修正し、統合テストについては責務を負わないので、システムのアップグレードには BFU を使用する方法がより一般的です。 カーネルで重点的に作業を行なっている開発者は、開発中に Install を活用することが多く、テスト段階で最新の状態に保つために BFU を使用します。

それぞれの方法については、このセクションで簡潔に説明します。 詳細の手順については、セクション 5.2 および 5.3 に記載します。

5.1.1 Cap-Eye-Install

Install (キャップアイ・インストールと発音。頭文字の I が大文字で記されるという意味) は、特定のシステムのカーネルと、そのカーネルに関連したモジュールを更新するときにのみ使用されます。 Install では、新しいカーネルが標準ではない場所に移され、特定のホスト向けのプラットフォーム固有モジュールのみがインストールされます。 これにより、通常のカーネルを削除せずに変更をテストすることができます。新しいカーネルがブートしなかったりクラッシュしたりしても、簡単に回復できます。

5.1.2 BFU

BFU は、すべての ON bits のカーネルとユーザランドを更新するために使用されます。 BFU は、一部の構成ファイルを更新でき、OpenSolaris に加えられた変更内容の影響も認識されます。 BFU は Install よりも綿密であり、時間も長くかかります。 さらに、Install とは異なり、新しいカーネルは既存のカーネルの上にインストールされるので、正常に動作しなければ代替メディアからブートして回復する必要があります。

5.1.3 Flag Day とその他の危険要因

場合によっては、 ON bits の新バージョンを使用する前に、新しいシステム・パッケージを 1 つ以上インストールしておく必要があります。 このような状況を Flag Day と呼びます。 Flag Day のお知らせは、[[http://opensolaris.org/os/community/on/>>#http___opensolaris_org_os_community_onnv_]] に投稿され、必要となる新規ソフトウェアの構築やインストールのための手順が記載されます。 このインストール手順は、pkgadd(1M) などの標準的なパッケージ・ツールとともに正常に動作することが保証されているわけではないことに注意してください。特に、BFU を使用したり (セクション 4.1.4 を参照)、Solaris コンポーネントを一時的に変更したりすると、これらのコンポーネントは Solaris パッケージを使用して更新できなくなります。 これが問題となる場合は、Solaris Express のみを使用してシステム・パッケージのインストールを管理し、最新の Express ビルドの時点で最新のソース・ツリーに対してのみビルドとテストを行うことをお勧めします。

5.2 Cap-Eye-Install を使用してカーネルをインストールする

まず、構築済みのカーネルを実装したワークスペースが必要になります。 カーネルの構築に関する詳細な説明が必要な場合は、第 4 章 (特にセクション 4.3) を参照してください。 カーネルを構築すると、Install コマンドを使用して Install tarball を作成する必要があります。

Install(1) ユーティリティは、適切なマシンのルート・ディレクトリに抽出できる tar ファイルを作成します。 このユーティリティは既存の構築済みカーネル・ツリーとカーネル makefile を使用し、tar ファイルの正しい内容を調べます。 完全なオプションのリストについては、Install(1) または「Install -h」の出力を参照してください。ここでは、もっとも頻繁に使用されているオプションのみを説明します。

Install によって構築された tar ファイルは、sun4u や i86pc などアーキテクチャ固有のものです。 Install によってアーカイブを作成するアーキテクチャを指定するには、2 通りの方法があります。 1 つめは、Install を実行中に、アーキテクチャのサブディレクトリ (usr/src/uts/<arch>) にいること、そして 2 つめは (推奨される方法) -k <arch> オプションを使用することです。 OpenSolaris の現行リリースでは、各 ISA に対して 1 つのアーキテクチャしかサポートしません。つまり、SPARC は sun4u、そして x86/amd64 は i86pc です。

そのほかに通常指定される設定は、-G オプションを使用する「glomname」です。 これは、バイナリが格納される /platform/<arch>/ のサブディレクトリ名であり、「kernel.something」の形式で表示されます。 glomname を使用しない場合は、ターゲット・マシンの現行カーネルとモジュールが上書きされ、後に行われるマシンの BFU が困難になる可能性があります。また、新しいカーネルがブートしなければ、代替メディアからブートしなければならないというリスクも高くなります。

次に、簡単な Install の起動例を示します。


        $ Install -G kernel.foo -k sun4u
        ... lots of spew ...
        Creating tarfile /tmp/Install.username/Install.sun4u.tar
        Install complete

これで、/tmp/Install.username/Install.sun4u.tar をテスト・マシンにコピーし、ルート・ディレクトリに抽出することができます。

ビルド 14 以降の x86 では、カーネルをブートする前に、ブート・アーカイブにカーネルを追加する必要があります。 これを行うには、


        platform/i86pc/kernel.foo

を /boot/solaris/filelist.ramdisk に追加する必要があります。ここで、kernel.foo は glomname になります (-Gの引数)。 この要件は、ビルド 18 以降では必要なく、SPARC にも適用されません。

カーネルをインストールすると、テスト・マシンを再起動し、次のように実行して新しいカーネルを使用します。


        (SPARC)
        # reboot ~-- 'kernel.foo/sparcv9/unix'


        (AMD64)
        # reboot ~-- 'kernel.foo/amd64/unix'


        (x86)
        # reboot ~-- 'kernel.foo/unix'

テスト・カーネルをブートするたびに、この再起動構文を使用するか、ブートローダまたは OBP に対する同様の引数を使用する必要があります。 これらを使用しなければ、BFU や通常のインストール手順でインストールされた通常のカーネルがブートされます。

5.2.1 注意

Install は、カーネル・コードのみを変更した開発者にとっては便利なものですが、それ以外にはあまり価値がありません。 特に、最近の flag day のお知らせに、新しいカーネルが既存のユーザランド・ライブラリやコマンドと適合しないことが記されている場合は、BFU や、通常のインストール手順またはアップグレード手順などによってユーザランドをアップグレードしなければ、Install を使用して更新されたカーネルをテストすることはできません。

bfu と同様 (セクション 5.3 を参照)、Install は特定のリリースに密接に関係しているので、構築しているリリースに適したゲートから最新バージョンを使用する必要があります。 通常、これはゲートの public/bin サブディレクトリにあります。ただし、Sun 以外のインストールについては、/opt/onbld/bin にあります。

特に SPARC システムについては、Install ユーザは正しいプラットフォーム固有モジュールをインストールする必要があります。 正しいプラットフォーム固有モジュールをインストールしなければ、システムがブートできなくなる場合があります。プラットフォーム固有モジュールが Install にどのように関連しているかについては、セクション 5.2.2 を参照してください。

BFU を使用する代わりに Install を使用することの主な利点は、既存のカーネルをそのまま残しておけるので、テスト・カーネルが異常であってもブートできることです。 Install を使用してカーネルをテストする場合は、この特長を活かし、テスト対象の新しいカーネルごとに別の場所 (上記のセクション 5.2.1、-G オプションに関する説明を参照) を使用することを強くお勧めします。 別の場所を使用しないと、不正なカーネルをインストールしたあとに、代替メディアからブートし、システムを復旧させなければならない可能性が高くなります。

5.2.2 プラットフォーム固有情報

通常、Install は実装固有のモジュールをもつアーカイブは生成しません。 これらのアーカイブが、欠落しているモジュールを必要とするシステムにインストールされると、システムはブートに失敗したり、正常に動作できなくなったりします。 このような場合には、動作することが分かっているカーネルを使用してブートし、問題を解決する必要があります。

次に、問題の現象例を示します (Sun Enterprise 3500 の場合)。


        SunOS Release on81 Version jrhyason_[ws-vmstat]_05/15/01 64-bit
        Copyright 1983-2001 Sun Microsystems, Inc.  All rights reserved.
        DEBUG enabled
        obpsym: symbolic debugging is available.
        Read 297063 bytes from misc/forthdebug
  ====>  WARNING: consconfig: consconfig_util_openvp failed: err 6 vnode
 0x2803c80
  ====>  WARNING: consconfig: consconfig_util_link I_PLINK failed: error
 22
        configuring IPv4 interfaces: hme0.
        ...

コンソールがなくなりました。

Install tarball に必要なモジュールを含めるには、


        $ Install -i <implementation>

を使用し、実装固有のモジュールを含める必要があります。 しかし、必要となる sun4u の実装がどれであるかは、どのようにして判断するのでしょうか。 まず、「uname -i」の出力から、マシンの「正式な」実装名を取得します。 次に、usr/src/uts/sun4u で「grep IMPLEMENTED_PLATFORM */Make*」を実行し、実装のリストと、uname(1) によってレポートされる、対応するプラットフォーム名を確認します。

上記の例では、E3500 は次のようにレポートします。


        $ uname -i
        SUNW,Ultra-Enterprise

grep の出力は次のとおりです。


        $ grep IMPLEMENTED_PLATFORM */Make*
        ...
        sunfire/Makefile:IMPLEMENTED_PLATFORM   = SUNW,Ultra-Enterprise
        ...

この場合、正しい動作を得るためには「-i sunfire」引数を必ず追加する必要があります。

さらに、Install で間違いやすい点は、すべてのドライブが ON によって配信されるわけではないことです。 これは特に x86 で顕著ですが、特にフレームバッファ・ドライバでは SPARC でも発生します。 これを回避するには、


 # cd /platform/{sun4u,i86pc}
 # mkdir glomname
 # (cd kernel; tar cf - .) | (cd glomname; tar xf -)

を実行し、Install glom イメージをインストールします。 これにより、既存のドライバが新しいカーネルの実装にコピーされるので、再起動時に ON (および OpenSolaris) の一部ではないドライバが利用可能になります。

5.3 BFU を使用して ON をインストールする

Blindingly Fast Upgrade (または Bonwick-Faulkner Upgrade) は、システム上の ON bits を更新するための手順です。 通常の Solaris 更新手順には完全な WOS が必要であり、少なくとも 30 分かかります (通常はそれ以上かかります)。 時間を節約するため、BFU では一連の cpio(1) アーカイブを使用し、システムの既存の内容を直接上書きします。 BFU は、完了までに通常は 10 分もかかりません。最新のビルドから更新する場合は、競合の解決も数分で終わります。 年間を通して使用すると、BFU は何十時間もの開発時間を短縮することができます。

BFU を使用するには、最初に 3 つの環境変数を追加で設定する必要があります。 これらの環境変数は、ログインのドットファイルかコマンド行で設定します。 環境変数を先に設定する bfu(1) のローカル・ラッパを作成することもできます。 環境変数は次のとおりです。

FASTFS
 これは、通常 /opt/onbld/bin/`uname -p`/fastfs に設定される必要があります。
BFULD
 これは、通常 /opt/onbld/bin/`uname -p`/bfuld に設定される必要があります。
GZIPBIN
 これは、通常 /usr/bin/gzip に設定される必要があります。

BFU は使いやすく、通常は単独の引数しか必要としません。 つまり、インストールする一連のアーカイブへのパスです。 たとえば、使用しているワークスペースが /home/jdoe/workspace にあり、nightly(1) のビルドを完了した場合は、次のようにして bfu(1) を起動します。


        # bfu /home/jdoe/workspace/archives/`uname -p`/nightly

これによってシステムのソフトウェア・インストールが修正されるので、bfu(1) は常に root で実行する必要があります。

bfu が完了すると、新規コマンドやライブラリと実行中の (既存の) カーネルとの互換性は保証されません。 したがって、bfu はシステムを終了する代わりに、サブシェル (PATH=/tmp/bfubin および LD_LIBRARY_PATH=/tmp/bfulib) に切り替えます。 これらのディレクトリには、通常、競合の解決とシステムの再起動に必要となる、旧バージョンのコマンドやライブラリが含まれます。また、古い動的リンカの保存されたコピーと動作できるよう修正されています。

「greenline.eng」またはその他のシステムや場所からファイルをコピーできないことを BFU から警告されることもあります。 通常、これらの警告はバグとしてレポートされる必要があります。 ただし、このドキュメントの作成時点では、BFU 試行の前にシステムが Solaris 10 ビルド 74 以降を実行している場合は、これらの警告から問題が生じることはありません。 追加要件および制限事項については、最新のリリース・ノートを参照してください。

5.3.1 注意

BFU は時間を節約することはできますが、すべての解決策ではありません。 このセクションでは、BFU の欠点について説明します。 これらの欠点を、利点に対して入念に評価し、BFU が各自のニーズに応えられるかどうかを判断する必要があります。 通常、現役の開発者以外には、BFU の使用をお勧めしません。

BFU はパッケージ情報を更新しません。 したがって、BFU で更新されたシステムに正式な Solaris パッチをインストールしたり、システムのフル・アップグレード手順を実行したりできなくなる可能性が高くなります。システムを BFU を使用して更新した場合は、今後、「通常の」システム管理手順によってシステムを更新せず、BFU を使用するか再インストールするということを絶対に忘れないでください。

BFU は、ON をインストールしたり、インストールされる ON のバージョンを実行するために ON 以外のパッケージの新しいバージョンが必要になった場合でも、これらのパッケージを更新しません。 これらのパッケージは、BFU の実行前または実行後に更新する必要があります。 どのパッケージの更新が必要であるかを理解するには、[[http://opensolaris.org/os/community/on/>>#http___opensolaris_org_os_community_onnv_]] で、現行のビルドと BFU を行うビルドとの間の flag day の完全なリストを確認してください。 flag day のお知らせには、必要となるパッケージ更新と (必要な場合のみ)、それを BFU の前に完了する必要があるかどうかが記されています。 BFU を実行する前に、これらの指示を読んで理解し、確実に従う必要があります。 これらを行わなければ、システムが ブリック化 (brickify) される可能性が高くなります。 flag day についての説明は、セクション 5.1.3 を参照してください。

BFU の中核的な機能は、実行中のシステムに cpio(1) アーカイブをアンパックするという単純な動作によって構成されますが、ほかにも、システムの更新に関する多数のタスクを実行します。 これらのタスクの多くは、数年にわたってソースに加えられた変更内容に固有のものです。 システムに例外的な性質があれば、これらの追加更新は失敗に終わり、システムは稼動しなくなります。 これらの更新内容はそれぞれに大きく異なるため、どのアップデートが失敗するか、また、どの障害モードが発生する可能性があるかを事前に把握することはできません。 このような障害は滅多に発生しませんが、可能性がないわけではありません。 BFU を使用すると、OpenSolaris の開発に従い、システムに悪影響を与える可能性のある変更内容を把握する必要があります。 OpenSolaris への大きな変更後に、BFU がシステムの特定の要素をどれほど効果的に更新できるかについて疑問がある場合は、bfu(1) を参照し、変更を加えたエンジニアに問い合わせてください。

bfu が終了すると、限定的な PATH をもつ ksh が起動されます。 PATH には、BFU アーカイブに含まれる変更内容にかかわらず動作するよう特別に修正されたプログラムが含まれます。 これらのプログラムを使用し、競合を解決するすることができます (セクション 5.3.2 を参照してください)。 具体的には、「reboot」は動作しますが「init 6」は動作しません。 必要であればこの保護環境を終了することはできますが、最後に bfu を実行してから「flag day」(同期カーネル / ユーザランドの変更) がなかったことに確信を持てない場合はお勧めしません。

また、BFU をウィンドウで絶対に実行しないでください。BFU は必ずシステム・コンソールを使用してください。 BFU またはシステムが途中でクラッシュした場合、またはウィンドウ・システムを使用しており、クラッシュやハングが発生した場合は、システムは不整合の状態になります。 この場合、ブートできない可能性があります。 したがって、BFU を開始する前にシステムの大部分が一時停止状態であること、必要に応じて再インストールできるよう、適切なシステム・ソフトウェア・メディアのコピーがあることを確認する必要があります。

本稼動システムで BFU を絶対に実行しないでください。本稼動システムは、サポートされるアップグレード機構を使用し、承認されたリリースに更新される必要があります。

5.3.2 競合を解決する

どのマシンにも、デフォルトのインストールから変更されるいくつかの構成ファイルがあります。bfu はこれらのファイルのリストを保持します。このリストは、競合データベースと呼ばれます。

BFU は、/bfu.child に各構成ファイルのコピーを保存します。同様に、以前の BFU の /bfu.parent (ある場合) を /bfu.ancestor に移動してから、 cpio アーカイブからこれらのファイルのコピーを作成し、/bfu.parent の下に抽出します。これらのファイルは、それぞれ子 (child)、親 (parent)、および祖先 (ancestor) としてディレクトリに保存します。

抽出が完了すると、BFU はこれらの構成ファイルのさまざまなバージョンを diff (差分抽出) を行い、次の規則に従って動作します。

  • ファイルが変更されていない場合は (つまり、子と親の間に相違がなければ)、実行したりレポートしたりするものはありません。
  • ファイルが、前回の実行時に親から承認された場合は (つまり、子が祖先と同じ)、今回は親が自動的に承認されます。このファイルは「update」としてマーク付けされます。
  • ファイルが前回のファイルの先頭と同じであれば、ユーザはファイルの最後に行を追加したことが推測されます (つまり、子は親と追加行で構成されます)。この場合、子のバージョンが復元されます。このファイルは「restore」とマークが付けられます。
  • 親と子の間でファイルの相違があり、親と祖先は同じであれば、過去の競合として見なされます。このようなファイルは「old」としてマークが付けられます。
  • 最後に、親と子の間でファイルの相違があり、親と祖先の間にも相違があれば、新しい競合として見なされます。このようなファイルは「NEW」としてマークが付けられます。

そのため、新規競合はデフォルトとは異なるファイル (デフォルトが更新されたもの) と見なすことができます。この相違点を解決するため、新規ビルドで導入された変更内容は、システム上の既存のファイルに移行する必要があります。

親を無計画に承認することも可能ですが、子に対して行われたカスタマイズが失われる原因となります。 これらのファイルは、ON 以外のパッケージからクラス・アクション・スクリプトによって自動的にカスタマイズされることがよくありますが、これは誤りであることが多く、何時間もの生産性の低下につながることもあります。 したがって、次の 2 つのセクションでは、2 つの競合解決方法について説明します。 自動で行う競合解決と、手動で行う競合解決です。

競合を手動で解決することを選択した場合、または自動ツールがすべての競合を解決できない場合は、適切な BFU ベースライン・インストールが存在すると便利です。 このようなベースラインを確立するには、ベースラインのインストール直後に、使用しているディストリビューションに対応する BFU アーカイブをインストールする必要があります。 このプロセスは、使用しているインストールが十分に新しく、このような BFU アーカイブが使用可能である場合にのみ開始することをお勧めします。 特に、Solaris 11 ビルド 16 以前のシステムで BFU を実行すると失敗し、システムがブートできなくなることがあります。

使用しているディストリビューションに含まれるビルドと同一のビルドに対して BFU を実行する場合は、すべての競合を無視することができます。これは、既存のインストール済み構成ファイルは、このビルドで正常に動作することが分かっているためです。 それ以降のビルドに対してBFU を実行する場合は、実行前に再起動する必要があります。

5.3.2.1 自動の競合解決

自動の競合解決は、acr と呼ばれるスクリプトによって実行されます。 このスクリプトは ON ワークスペースの /opt/onbld/bin および usr/src/tools/scripts ディレクトリにあります。 usr/src/tools/scripts ディレクトリでは、acr 実行ファイルは -t オプションとともに nightly(1) ビルドを実行すると、acr.sh から作成されます。 ON ワークスペースの /opt/onbld/bin および usr/src/tools/scripts ディレクトリに、マニュアル・ページ (acr,1) があります。

acr を使用する標準的な方法とは、BFU の実行後に保護環境で起動することです。 このモードでは、コマンド行パラメータを指定する必要はありません。

特殊なアプリケーションの場合、acr は 1 つまたは 2 つのパラメータを使用します。 1 つめのパラメータは、代替ルート・ディレクトリ名です。 2 つめのパラメータは (指定される場合)、アーカイブを含むディレクトリ名です。

acr は、BFU cpio アーカイブを含むディレクトリの conflict_resolution.gz というファイルを使用します。 この競合解決ファイルは、nightly(1) がアーカイブを作成するたびに構築されます。 つまり、-a オプションを使用して nightly(1) が実行されるたびに構築されるということです。

acr が実行されると、標準出力のために処理するファイルをリストします。 詳細の結果は、/tmp のサブディレクトリにあるファイル allresults に書き込まれます。 acr が終了すると、allresults ファイルのフル・パス名が出力されます。 acr の処理中にエラーが発生していないかどうかを確認するため、allresults を検証する必要があります。 エラーが発生した場合は、次のセクションで説明する手動の競合解決手順を実行する必要があります。

5.3.2.2 手動の競合解決

手動の競合解決は、競合を一つ一つ手作業で解決する必要があります。 競合を解決するには、一般的に


        % diff /bfu.ancestor/$file /bfu.parent/$file

を実行し、/$file に diff コマンドを実行します。 前に BFU を実行していない場合、 /bfu.ancestor はありません。したがって、前述のようにベースラインを確立しておけば、手動の競合解決が簡単になります。多くのファイルでは、ショートカットを利用できます。 BFU が完了すると、ユーザは /bfu.conflicts にドロップされ、変更内容の大半は追加や変更になります。そのため、


        % diff $file /$file

を実行し、差分がすべて右向きの矢印であれば (つまり「<」ではなく「>」)、問題なく処理されます。 ただし、これがすべてのケースにおいて有効なわけではありません。次に、主な例外を示します。

etc/name_to_major

このファイルは、デバイス名とメジャー番号を照合します。各デバイスに、一意のメジャー番号が割り当てられている必要があります。 まずは、祖先を親とを diff で比較すると良いでしょう。ただし、名前が重要であることは覚えておいてください。メジャー番号はそれぞれ異なります。 親から消去された古いデバイスは削除する必要があります。ただし、確かではない場合は、そのまま残しておいても問題となることはありません。 新しいデバイスは、未使用のメジャー番号を割り当てた上で追加する必要があります (可能であれば、diff 出力からの番号を使用します)。 既存のエントリの番号については、何がどのように処理されているかを明確に把握している場合を除き、絶対に変更しないでください。 何が変更されたかが分からない場合は、ワークスペース から直接ファイルの履歴を確認することができます (usr/src/uts/{intel|sparc}/os/name_to_major など)。

終了したら、次のように妥当性検査を実行します。


        % sort -k1,1 /etc/name_to_major | sort -uc -k1,1
        % sort -k2n /etc/name_to_major | sort -uc -k2n

これらは、それぞれ最初に重複したデバイス名と、重複されたメジャー番号のうちもっとも低いものをレポートlします。 これらのいずれも、再起動の前に解決しなければならない問題を示すわけではありません。 出力がなければ問題はありません。 再起動後に、カーネルがこのような競合を警告することもありますが、そのときにはすでに遅すぎることがあります。

etc/security/*_attr

これらのファイルは、クラスアクション・スクリプトによって大幅に変更されていることが多いので、親と子のバージョンも大きく異なることがあります。 このような場合、前述のショートカットは無効になりますが、親と祖先との一般的な diff は問題なく動作します。

一般的に、新しい競合のみ検証を必要とします。

5.3.3 BFU とゾーン

各ゾーンの内容は、ゾーンが作成されたときにグローバル・ゾーンからコピーされます。 すべてのゾーンが BFU の実行後も稼動可能な状態であるために、ゾーンの内容とグローバル・ゾーンの整合性を保つ必要があります。 そのために、グローバル・ゾーンが作成されると、BFU は各ゾーンを順番に更新します。ただし、各ゾーンにはグローバル・ゾーンとは異なる構成ファイルがある場合があるので、各ゾーンには独自の BFU 競合管理ディレクトリがあります。 残念ながら、単独のゾーンを対象に BFU を実行することはできません。 BFU は、すべてのゾーンを含むシステム全体に影響を及ぼします。 ゾーン内のユーザ空間への変更をテストするには、プロト・エリアからそのゾーンにファイルを手動コピーする必要があります。

システムに対して BFU を実行する前に、すべてのゾーンをシャットダウンすることがもっとも安全です。 シャットダウンしておくことにより、ファイルをインストールすることで依存関係が侵害されても、ゾーンが破壊されずに済みます。

グローバル・ゾーン以外のゾーンをもつシステムに対して BFU を実行する場合は、まずグローバル・ゾーンの競合を解決し、その後残りの各ゾーンの競合を解決してから、システムを再起動する必要があります。 ゾーン内の BFU 競合解決は、グローバル・ゾーンとまったく同じ方法で行われます。ただし、多くの場合、ファイルの内容を手動で結合する必要はありません。 ファイルがグローバル・ゾーンのファイルとまったく同じであることが多いので、単に上書きコピーすることができます。 すべてのゾーンの競合を解決した後は、ゾーンを再起動する前にシステムを再起動する必要があります。

5.4 テスト・スイート

Sun では、ON で使用できるテスト・スイートをいくつか提供しています。 残念ながら、これらは現在のところまだ利用できません。 [[http://opensolaris.org/os/about/roadmap/>>#http___opensolaris_org_os_about_roadmap_]]のロードマップを確認してください。
