Perguntas de versão do ZFS

  1. Com obtenho o ZFS?
  2. Quando o ZFS estará disponível para <inserir SO aqui>
  3. O que quer dizer ZFS?
  1. Com obtenho o ZFS?
    O ZFS está disponível nas seguintes versões:
    1* Versão Solaris Nevada, build 27a e posteriores
    1* Versões Solaris Express
    1* A partir da versão 10 6/06 do Solaris
  2. Quando o ZFS estará disponível para <inserir SO aqui>
    Existem projetos em andamento para a portagem do ZFS para o FreeBSD e o Linux (usando o FUSE). Para obter mais informações sobre o CDDL, consulte a FAQ sobre licenciamento.
  3. O que quer dizer ZFS?
    Originalmente, o ZFS era sigla de "Zettabyte File System". O maior prefixo SI de que gostamos foi 'zetta' ('yotta' estava fora de questão). Como ZFS é um sistema de arquivos de 128 bits, o nome faz referência ao fato de que o ZFS pode armazenar 256 quadrilhões de zettabytes (onde cada ZB tem 270 bytes). Com o tempo, o ZFS ganhou muitos outros recursos além da capacidade de 128 bits, como integridade de dados sólida, fácil adminsitração e um modelo simplificado de gerenciamento de dados.

Perguntas técnicas sobre o ZFS

  1. Por que o ZFS tem capacidade de 128 bits?
  2. Quais são os limites do ZFS?
  1. Por que o ZFS tem capacidade de 128 bits?
     Os sistemas de arquivos provaram ter um ciclo de vida muito maior do que aquele da maioria dos softwares, em parte devido ao fato de que o formato em disco é extremamente difícil de alterar. Dado o fato de que o UFS tem perdurado em sua forma atual (em grande parte) há quase 20 anos, é razoável esperar que o ZFS dure pelo menos 30 anos no futuro. Nesse ponto, a lei de Moore começa a atuar no armazenamento e começamos a prever que armazenaremos mais de 64 bits de dados em um único sistema de arquivos. Para uma descrição mais completa desse tópico e do motivo pelo qual 128 bits são suficientes, consulte aentrada de blog do Jeff
  2. Quais são os limites do ZFS?
    As limitações do ZFS foram criadas para serem tão grandes que nunca serão encontradas em nenhuma operação prática. O ZFS pode armazenar 16 Exabytes em cada pool de armazenamento, sistema de arquivo, arquivo ou atributo de arquivo. O ZFS pode armazenar bilhões de nomes: arquivos ou diretórios em um diretório, sistemas de arquivos em um sistema de arquivos, ou snapshots de um sistema de arquivos. O ZFS pode armazenar trilhões de itens: arquivos de um sistema de arquivos, sistemas de arquivos, volumes ou snapshots de um pool.

Perguntas de comparação entre o ZFS e o UFS

  1. Por que o ZFS não tem um utilitário como o fsck?
  2. Por que o du(1) reporta tamanhos de arquivos diferentes para o ZFS e o UFS?
  3. Por que o ZFS não tem cotas de usuário ou grupo?
  1. Por que o ZFS não tem um utilitário como o fsck?
     Existem dois motivos básicos para ter um utilitário como o fsck:
    1* Verificar a integridade do sistema de arquivos - muitas vezes, os administradores simplesmente querem ter certeza de que não há corrupção de disco dentro de seus sistemas de arquivos. Na maioria dos sistemas de arquivos, isso envolve a execução do fsck enquanto o sistema de arquivos está offline. Isso pode ser demorado e caro. Em vez disso, o ZFS oferece a capacidade de 'limpar' todos os dados de um pool enquanto o sistema está ao vivo, encontrando e reparando os dados que apresentam problemas. Existem planos futuros para ampliar isso e permitir a 'limpeza' em segundo plano.
    1* Reparar o estado em disco - se uma máquina dá pane, o estado em disco de alguns sistemas de arquivos é inconsistente. A adição do journalling solucionou alguns desses problemas, mas a falta de restabelecimento do log ainda pode resultar em um sistema de arquivos que precisa ser reparado. Neste caso, existem conhecidas patologias de erros, como a criação de uma entrada de diretório antes da atualização do link pai, o qual pode ser reparado de forma confiável. O ZFS não sofre desse problema, porque os dados estão sempre consistentes no disco.
    Um problema mais insidioso ocorre com hardware ou software com falhas. Mesmo os sistemas de arquivos ou os gerenciadores de volume que têm checksums por bloco são vulneráveis a uma variedade de outras patologias que resultam em dados válidos, mas corrompidos. Neste caso, o modo de falha é essencialmente aleatório, e a maioria dos sistemas de arquivos entra em pânico (caso sejam metadados), ou retornam silenciosamente dados ruins para o aplicativo. Em ambos os casos, um utilitário fsck será de pouca ajuda. Como a corrupção não coincide com nenhum patologia conhecida, provavelmente ela será irreparável. Com o ZFS, esses erros existirão (estatisticamente) em uma configuração redundante. Em uma configuração não redundante, esses erros são detectados corretamente, mas resultam em um erro de E/S ao tentar ler o bloco. Em teoria, é possível escrever uma ferramenta para reparar tal corrupção, embora tal tentativa provavelmente seria uma ferramenta especial on-off. Obviamente, o ZFS é igualmente vulnerável a bugs de software, mas os bugs teriam que resultar em um padrão consistente de corrupção para serem reparados por uma ferramenta genérica. Durante os 5 anos de desenvolvimento do ZFS, nenhum desses padrões foi visto.
  2. Por que o du(1) reporta tamanhos de arquivos diferentes para o ZFS e o UFS?
    No UFS, o du(1) reporta o tamanho dos blocos de dados dentro do arquivo. No ZFS, o du(1) reporta o tamanho atual do arquivo armazenado em disco. Esse tamanho inclui os metadados, bem como a compactação. Esse relatório realmente ajuda a responder a pergunta "quanto espaço a mais vou obter se remover esse arquivo?" Assim, mesmo quando a compactação está desligada, você ainda vê resultados diferentes entre o ZFS e o UFS.
  3. Por que o ZFS não tem cotas de usuário ou grupo?
    Os sistemas de arquivo ZFS podem ser usados como pontos de controle administrativo lógicos, os quais permitem visualizar o uso, gerenciar propriedades, realizar backups, fazer snapshots e outros. Para os servidores de diretórios principais, o modelo ZFS permite configurar facilmente um sistema de arquivos por usuário. As cotas do ZFS não estão associadas a um usuário em particular intencionalmente, porque os sistemas de arquivos são pontos de controle administrativo.
    As cotas do ZFS podem ser definidas nos sistemas de arquivos que representariam usuários, projetos, grupos e outros, bem como em partes inteiras de uma hierarquia de sistema de arquivos. Isso permite que as cotas sejam combinadas de maneiras que seriam impossíveis para as cotas por usuário tradicionais. As cotas por usuário foram introduzidas porque vários usuários tinham que compartilhar o mesmo sistema de arquivos.
    As cotas do sistema de arquivos ZFS são flexíveis e fáceis de configurar. Uma cota pode ser aplicada quando o sistema de arquivos é criado. Por exemplo:
# zfs create -o quota=20g tank/home/users

Os sistemas de arquivos de usuário neste sistema de arquivos herda automaticamente a cota de 20 Gbyte definida no sistema de arquivos pai. Por exemplo: |

# zfs create tank/home/users/user1
# zfs create tank/home/users/user1
# zfs list -r tank/home/users
NAME                    USED  AVAIL  REFER  MOUNTPOINT
tank/home/users        76.5K  20.0G  27.5K  /tank/home/users
tank/home/users        76.5K  20.0G  27.5K  /tank/home/users
tank/home/users        76.5K  20.0G  27.5K  /tank/home/users

As cotas do ZFS podem ser aumentadas quando o espaço em disco dos pools de armazenamento do ZFS aumentam enquanto os sistemas de arquivos estão ativos, sem necessidade de paralisação.
Em vez de tentar fazer com que as cotas baseadas em usuário se ajustem a um modelo de administração baseado em sistema de arquivos como pontos de controle, a equipe do ZFS está trabalhando para melhorar o gerenciamento de vários sistemas de arquivos.
Uma alternativa para as cotas baseadas em usuário para conter o espaço em disco usado para mensagens é o uso do software de servidor de mensagens que inclui um recurso de cotas, como o Sun Java System Messaging Server. Esse software fornece cotas de mensagens de usuário, mensagens de aviso de cota e recursos de vencimento e expurgo.

Perguntas de administração do ZFS

  1. O que posso fazer se o ZFS entrar em pânico a cada inicialização?
  2. O ZFS suporta hot spares?
  3. Os dispositivos podem ser removidos de um pool do ZFS?
  4. Eu posso usar o ZFS como meu sistema de arquivos raiz? E quanto às zonas?
  5. Eu posso dividir uma configuração ZFS mirrored?
  1. O que posso fazer se o ZFS entrar em pânico a cada inicialização?
    O ZFS foi criado para sobreviver a falhas de hardware arbitrárias usando a redundância (mirroring ou RAID-Z). Infelizmente, determinadas falhas em configurações não replicadas podem fazer com que o ZFS entre em pânico ao tentar carregar o pool. Isso é um bug que será corrigido em um futuro próximo (juntamente com vários outros recursos, como a limpeza em segundo plano). Nesse meio tempo, se você não conseguir inicializar devido a um pool corrompido, faça o seguinte:
    1. inicialize usando '-m milestone=none'
    2. # mount -o remount /
    3. # rm /etc/zfs/zpool.cache
    4. # reboot
       Isso removerá todo o conhecimento dos pools de seu sistema. Você terá que recriar seu pool e restaurar do backup.
  2. O ZFS suporta hot spares?
    Sim, o recurso hot spares do ZFS está disponível no Solaris Express Community Release, build 42, versão Solaris Express julho de 2006 e versão Solaris 10 11/06. Para obter mais informações sobre os hot spares, consulte o Guia de administração do ZFS.
  3. Os dispositivos podem ser removidos de um pool do ZFS?
    No momento, a remoção de um vdev de primeiro nível não é suportada, como todo um grupo RAID-Z ou um disco em uma configuração sem mirroring. Esse recurso está planejado para uma versão futura.
    Você pode remover um dispositivo de uma configuração ZFS mirrored usando o comando zpool detach.
    Você pode substituir um dispositivo por outro com tamanho equivalente em uma configuração mirrored ou RAID-Z, usando o comando zpool replace.
  4. Eu posso usar o ZFS como meu sistema de arquivos raiz? E quanto às zonas?
    No momento, os sistemas de arquivos ZFS não podem ser usados em um sistema de arquivos raiz nas versões do Solaris 10. Entretanto, um pequeno subconjunto de suporte a inicialização e raiz do ZFS está disponível na versão da comunidade SX para os sistemas x86. Para obter outras informações, consulte Inicialização do ZFS. Permaneça informado sobre a programação de inicialização do Solaris 10 ZFS.
    O ZFS pode ser usado como um caminho de raiz de zona na versão Solaris Express, mas não pode ser corrigido ou atualizado antes que essas ferramentas reconheçam os sistemas de arquivos do ZFS. Para obter outras informações, consulte a FAQ de zonas.
    Além disso, você não pode criar um cache cachefs em um sistema de arquivos ZFS.
  5. Eu posso dividir uma configuração ZFS mirrored?
    No momento, o ZFS não suporta a capacidade de dividir uma configuração mirrored para finalidades de clonagem ou backup. O melhor método para clonagem e backups é usar os recursos de clonagem e snapshot do ZFS. Para obter informações sobre como usar os recursos de clonagem e snapshot do ZFS, consulte o Guia do admnistrador do ZFS. Consulte o RFE 6421958 para enviar recursivamente snapshots que melhorarão o processo de replicação nos sistemas.
    Além dos recursos de clonagem e snapshot do ZFS, a replicação remota dos sistemas de arquivos ZFS é fornecida pelo produto Sun StorageTek Availability Suite. As demonstrações do AVS/ZFS estão disponíveis aqui.
    Lembre-se destas advertências ao tentar dividir uma configuração do ZFS mirrored para fins de clonagem ou backup:
    1* A divisão de uma configuração do ZFS mirrored não é suportada pelo ZFS. O RFE 6421958 é registrado para fornecer esse recurso.
    1* Você não pode remover um disco de uma configuração do ZFS mirrored, nem pode fazer o backup dos dados em disco e depois usar esses dados para criar um pool clonado.
    1* Caso queira usar o backup no nível de hardware ou o recurso de snapshot em vez do recurso de snapshot do ZFS, você tem que seguir as seguintes etapas:
      1. zpool export pool-name
      2. Etapas para o snapshot no nível de hardware
      3. zpool export pool-name
        1* Qualquer tentativa de dividir um pool de armazenamento do ZFS mirrered, remover discos ou alterar o hardware que faz parte de um pool ao vivo causaria corrupção de dados.

Perguntas sobre a interação entre o ZFS e outros produtos

  1. O ZFS é suportado em um ambiente de cluster?
  2. Quais produtos de backup de terceiros suportam o ZFS?
  3. O ZFS funciona com dispositivos anexados ao SAN?
  1. O ZFS é suportado em um ambiente de cluster?
    O Solaris Cluster 3.2 suporta um sistema de arquivos ZFS local como altamente disponível (HA) na versão Solaris 10 11/06. Esse suporte permite o failover ao vivo entre sistemas, com importação automática de pools entre os sistemas.
    Caso você use o Solaris Cluster 3.2 para configurar um sistema de arquivos ZFS local como altamente disponível, examine esta advertência:
    Não adicione um dispositivo de quórum configurado a um pool de armazenamento do ZFS. Quando um dispositivo de quórum configurado é adicionado a um pool de armazenamento, o disco é rotulado novamente e as informações de configuração de quórum se perdem. Isso significa que o disco não fornece mais um voto de quórum ao cluster. Após a adição de um disco em um pool de armazenamento, você pode configurar esse disco como um dispositivo de quórum. Ou então você pode desconfigurar o disco, adicioná-lo ao pool de armazenamento e depois reconfigurar o disco como um dispositivo de quórum.
    O Solaris Cluster 3.2 não é suportado nas versões OpenSolaris ou Nevada. Para obter informações sobre o uso da versão Solaris Cluster de fonte aberta, visite a página da comunidade de cluster de alta disponibilidade aberta.
    O ZFS não é um sistema de arquivos de cluster nativo, distribuído ou paralelo e não fornece acesso simultâneo de vários hosts diferentes. O ZFS funciona muito bem quando é compartilhado em um ambiente NFS distribuído.
    No longo prazo, planejamos investigar o ZFS como um sistema de arquivos de cluster nativo para permitir o acesso simultâneo. A extensão desse trabalho ainda não foi avaliada.
  2. Quais produtos de backup de terceiros suportam o ZFS?
    1* O EMC Networker 7.3.2. faz o backup e restaura os sistemas de arquivos do ZFS, incluindo as ZFS ACLs.
    1* O Veritas Netbackup 6.5 faz o backup e restaura os sistemas de arquivos do ZFS, incluindo as ZFS ACLs.
    1* O software de cliente IBM Tivoli Storage Manager (5.4.1.2) faz o backup e restaura os sistemas de arquivos do ZFS com a CLI e a GUI. As ZFS ACLs também são preservadas.
    1* O produto BrightStor ARCserve da Computer Associates faz o backup e restaura os sistemas de arquivos do ZFS, mas as ZFS ACLs não são preservadas.
  3. O ZFS funciona com dispositivos anexados ao SAN?
    Sim, o ZFS funciona com dispositivos diretamente anexados ou com dispositivos anexados pelo SAN. Entretanto, se o seu pool de armazenamento não contiver dispositivos de mirror ou de primeiro nível RAID-Z, o ZFS só pode reportar os erros de checksum, mas não pode corrigi-los. Caso seu pool de armazenamento consista de dispositivos de mirror ou RAID-Z criados usando armazenamento de dispositivos anexados pelo SAN, o ZFS pode reportar e corrigir os erros de checksum.
    Por exemplo, pense em um array RAID de hardware anexado pelo SAN, configurado para apresentar LUNs para o SAN fabric que se baseiam em seus discos mirrored internamente. Caso você use um único LUN desse array para criar um pool de disco simples, o pool não conterá os dados em duplicata que o ZFS precisar para corrigir os erros detectados. Neste caso, o ZFS não poderia corrigir um erro introduzido pelo array.
    Caso você use dois LUNs desse array para construir um pool de armazenamento mirrored, ou três LUNs para criar um pool de armazenamento RAID-Z, o ZFS teria dados em duplicata para corrigir os erros detectados. Neste caso, o ZFS poderia corrigir os erros introduzidos pelo array.
    Em todos os casos onde os pools de armazenamento do ZFS não têm dispositivos de mirror ou dispositivos virtuais de nível principal RAID-Z, a viabilidade do pool depende totalmente da confiabilidade dos dispositivos básicos de armazenamento.
    Caso seu pool de armazenamento do ZFS contenha apenas um único dispositivo, seja anexado pelo SAN ou anexado diretamente, você não pode aproveitar recursos como RAID-Z, striping dinâmico, balanceamento de carga de E/S e outros.
    O ZFS sempre detecta a corrupção de dados silenciosa. Alguns arrays de armazenamento podem detectar erros de checksum, mas não seriam capazes de detectar as seguintes classes de erros:
    1* Sobregravações acidentais ou gravações fantasmas
    1* Leituras e gravações mau direcionadas
    1* Erros de caminho de dados
    Em geral, as funções do ZFS são criadas com dispositivos anexados pelo SAN, mas se você expuser dispositivos mais simples ao ZFS, poderá aproveitar melhor todos os recursos disponíveis.
    Em resumo, se você usar o ZFS com dispositivos anexados pelo SAN, você poderá aproveitar os recursos de auto cura do ZFS, configurando a redundância em seus pools de armazenamento do ZFS, embora a redundância esteja disponível em um nível de hardware inferior.
