ZFS 製品リリースに関する質問

  1. ZFS の入手先を教えてください
  2. <ここに OS 名を挿入> では、いつから ZFS が使用できるようになりますか
  3. ZFS とは何の略語ですか
  1. ZFS の入手先を教えてください
    ZFS は次の各リリースに同梱されています。
    1* Solaris Nevada リリース、ビルド 27a 以降
    1* Solaris Express リリース
    1* Solaris 10 6/06 リリース以降
  2. <ここに OS 名を挿入> では、いつから ZFS が使用できるようになりますか
    ZFS を FreeBSD および Linux (FUSE を使用) に移植するプロジェクトが進行中です。 CDDL については、「ライセンスに関する FAQ」を参照してください。
  3. ZFS とは何の略語ですか
    もともと、ZFS は "Zettabyte File System" の頭文字を使った略語でした。 候補となった SI 接頭辞のうちで最大のものが 'zetta' ('yotta' は問題外) でした。 ZFS は 128 ビットファイルシステムであるため、25 京 6 千兆 Z バイト (Zettabyte。1Z バイトは 270 バイト) までを格納できるという事実を表しています。 やがて、ZFS は 128 ビットの演算能力以外にも多数の機能を備えるようになりました。たとえば、堅牢なデータ整合性、管理の容易さ、および単純化されたデータ管理モデルなどがあります。

ZFS についての技術的な質問

  1. なぜ ZFS には 128 ビットの演算能力があるのですか
  2. ZFS にはどのような制限がありますか
  1. なぜ ZFS には 128 ビットの演算能力があるのですか
     ディスクの書式はなかなか変更しがたいという事実も手伝って、ファイルシステムは通常のソフトウェアよりも長い寿命を持つことが立証されています。 UFS がほぼ現在の形で約 20 年も使用され続けてきたという事実に鑑みれば、ZFS が少なくとも向こう 30 年にわたって使用され続けると期待するのは非合理的なことではありません。 現在、ムーアの法則がストレージにも該当しつつあり、Sun では 64 ビットを超えるデータを単一のファイルシステムに格納するようになると予測し始めています。 このトピック、および 128 ビットで十分な理由については、「Jeff のブログエントリ」を参照してください。
  2. ZFS にはどのような制限がありますか
    ZFS の制限は非常に大きな規模で設計されているため、実際の操作で目にすることはないでしょう。 ZFS は各ストレージプール、ファイルシステム、ファイル、またはファイル属性に 16E バイト (Exabyte) まで格納できます。 ZFS には数十億の名前を格納できます。 すなわち、単一ディレクトリ内の各ファイルやディレクトリ、単一ファイルシステム内の各ファイルシステム、ファイルシステムの各スナップショットなどの名前です。 さらに、 単一ファイルシステム内の各ファイル、単一プール内のファイルシステム、ボリューム、スナップショットなどの各項目については、数兆個を格納できます。

ZFS/UFS の比較に関する質問

  1. なぜ ZFS には fsck のようなユーティリティーがないのですか
  2. du(1) が ZFS と UFS とで異なるファイルサイズを報告するのはなぜですか
  3. ZFS にユーザーやグループの割り当て制限がないのはなぜですか
  1. なぜ ZFS には fsck のようなユーティリティーがないのですか
     fsck のようなユーティリティーが必要となる根本的理由としては、次の 2 つが挙げられます。
    1* ファイルシステムの整合性の確認 - 管理者は、ディスク上のファイルシステムが破損することだけは避けたいとしばしば思うものです。 そのためには、ほとんどのファイルシステムの場合、オフライン時に fsck を実行する必要があります。 ただし、この作業は時間もコストもかかります。 これに代わるものとして、ZFS ではシステムが稼働中にプール内のすべてのデータをスクラブすることができ、その処理中に不良データを検索して修復します。 今後はこの機能を拡張して、バックグラウンドでスクラブを実行できるようにする予定です。
    1* オンディスク状態の修復 - マシンがクラッシュすると、一部のファイルシステムのオンディスク状態で矛盾が発生します。 ジャーナル機能が追加されたことでこれらの問題のいくつかが解決されましたが、ログをロールフォワードできないためにファイルシステムの修復が依然として必要になる場合があります。 この場合、確実に修復できるエラー (たとえばリンク元を更新する前にディレクトリエントリを作成するなど) についてのよく知られた病理学があります。 ZFS では、ディスク上のデータには常に一貫性があるため、こうした問題は発生しません。
    ハードウェアやソフトウェアに問題がある場合は、さらに油断のならない問題が発生します。 ブロックごとのチェックサムを持つファイルシステムやボリュームマネージャーさえ、それ以外のさまざまな病理には弱いため、データは有効だが破損しているという事態に陥ります。 この場合、障害の形態は基本的にランダムで、特にメタデータに問題があると、ほとんどのファイルシステムがパニック状態になるかまたは不良なデータをそのままアプリケーションに返します。 どちらの場合でも、fsck ユーティリティーはほとんど役に立ちません。 このような破損の病理はまだ明らかになっていないため、修復できない可能性があります。 ZFS では、これらのエラーは冗長構成では統計上存在しません。 非冗長構成では、これらのエラーは正しく検出されるものの、ブロックを読み取ろうとすると I/O エラーになります。 このような破損を修復するツールは、理論的には作成可能ですが、そのエラー専用のツールになると考えられます。 もちろん、ZFS も同様にソフトウェアのバグには影響を受けますが、バグによる破損を一般的なツールで修復するには、破損のパターンに一貫性があることが必要です。 ZFS 開発の 5 年間で、そのようなパターンは見つかっていません。
  2. du(1) が ZFS と UFS とで異なるファイルサイズを報告するのはなぜですか
    UFS では、du(1) はファイル内のデータブロックのサイズを報告します。 一方、ZFS では、du(1) はディスクに格納されているファイルの実際のサイズを報告します。 このサイズにはメタデータや圧縮データも含まれています。 この情報によって、「このファイルを削除したらどれくらいのスペースが空くのか」という疑問に対する答えが得られます。 圧縮機能がオフの場合でも、ZFS と UFS では異なる結果が表示されます。
  3. ZFS にユーザーやグループの割り当て制限がないのはなぜですか
    ZFS ファイルシステムは、使用状態の表示、プロパティーの管理、バックアップの実行、スナップショットの取得などを行うことができる、管理上の論理的な制御点として使用できます。 ZFS モデルは、ホームディレクトリサーバーではユーザーごとに 1 つのファイルシステムを簡単に設定することができます。 ファイルシステムは管理上の制御点のため、ZFS の割り当て制限が特定のユーザーに割り当てられていないのは意図的なものです。
    ZFS 割り当て制限はユーザー、プロジェクト、グループなどを表すファイルシステムに対して設定することも、ファイルシステム階層の構成要素全体に対して設定することもできます。 このため、従来のユーザー別割り当て制限ではできなかった方法で、複数の割り当て制限を組み合わせることができます。 ユーザー別割り当て制限が導入された理由は、複数のユーザーが同じファイルシステムを共有する必要があったためです。
    ZFS ファイルシステムの割り当て制限は柔軟性があり、簡単に設定できます。 割り当て制限は、ファイルシステムの作成時に適用されます。 たとえば、次のように指定します。
# zfs create -o quota=20g tank/home/users

このファイルシステムに作成されるユーザーファイルシステムは、親のファイルシステムに設定されている 20G バイトの割り当て制限を自動的に継承します。 たとえば、次のように指定します。 |

# zfs create tank/home/users/user1
# zfs create tank/home/users/user2
# zfs list -r tank/home/users
NAME                    USED  AVAIL  REFER  MOUNTPOINT
tank/home/users        76.5K  20.0G  27.5K  /tank/home/users
tank/home/users/user1  24.5K  20.0G  24.5K  /tank/home/users/user1
tank/home/users/user2  24.5K  20.0G  24.5K  /tank/home/users/user2

ZFS の割り当て制限は、ZFS ストレージプール内のディスク容量が追加されたときに、停止時間を発生させることなくファイルシステムの稼働中に増加させることができます。
ZFS チームでは、制御点としてのファイルシステムをベースとした管理モデルにユーザー別割り当て制限を適合させようとするのでなく、複数のファイルシステム管理を改良しようと努力しています。
メール用のディスク容量を含むユーザー別割り当て制限の代替手段は、Sun Java System Messaging Server など、割り当て制限機能を持つメールサーバーソフトウェアを使用することです。 このソフトウェアには、ユーザーメール割り当て制限、割り当て制限警告メッセージ、期限設定および削除機能が用意されています。

ZFS の管理に関する質問

  1. 起動のたびに ZFS がパニック状態になったらどうすればよいですか
  2. ZFS ではホットスペアはサポートされていますか
  3. ZFS プールからデバイスを削除できますか
  4. ZFS をルートファイルシステムとして使用することはできますか。 また、ゾーンについてはどうですか
  5. ミラー化された ZFS 構成を分割できますか
  1. 起動のたびに ZFS がパニック状態になったらどうすればよいですか
    ZFS は、ミラー化や RAID-Z などの冗長性を利用することで、どのようなハードウェア障害が発生しても破損しないように設計されています。 残念ながら、複製されていない構成では、一部の障害が発生した場合に、ZFS はプールを読み込む際にパニック状態になることがあります。 これはバグで、近い将来に修正される予定です。その際、バックグラウンドでのスクラブの実行など、その他いくつかの優れた機能も登場する予定です。 それまでの間、プールが破損したために起動できなくなった場合は、次のコマンドを実行してください。
    1. boot using '-m milestone=none'
    2. # mount -o remount /
    3. # rm /etc/zfs/zpool.cache
    4. # reboot
       これにより、プールのすべてのデータがシステムから削除されます。 プールを作成し直し、バックアップから復元してください。
  2. ZFS ではホットスペアはサポートされていますか
    はい、ZFS ホットスペア機能は Solaris Express コミュニティーリリース、ビルド 42、Solaris Express 2006 年 7 月リリース、および Solaris 10 11/06 リリースで使用可能です。 ホットスペアについては、『ZFS 管理ガイド』を参照してください。
  3. ZFS プールからデバイスを削除できますか
    RAID-Z グループ全体や未ミラー化ディスクなど、最上位の vdev を削除することは現時点ではサポートされていません。 この機能は将来のリリース用に計画されています。
    ミラー化された ZFS 構成の場合は、zpool detach コマンドを使用してデバイスを削除できます。
    ミラー化された構成および RAID-Z 構成では、zpool replace コマンドを使用して、デバイスを同サイズのデバイスに交換できます。
  4. ZFS をルートファイルシステムとして使用することはできますか。 また、ゾーンについてはどうですか
    現在のところ、Solaris 10 リリースで ZFS をルートファイルシステムとして使用することはできません。 ただし、ZFS ルートの小さいサブセットおよびブートサポートは、x86 システム版 SX コミュニティーリリースで利用できます。 詳細は、ZFS ブートを参照してください。 Solaris 10 ZFS ブートの今後の展開にご期待ください。
    Solaris Express リリースでは ZFS をゾーンルートパスとして使用できますが、パッチの適用やアップグレードは、これらのツールが ZFS ファイルシステムを認識したあとでないと実行できません。 詳細は、ゾーンに関する FAQ を参照してください。
    また、ZFS ファイルシステムで cachefs キャッシュを作成することはできません。
  5. ミラー化された ZFS 構成を分割できますか
    現在のところ、ZFS では、複製またはバックアップの目的でミラー化された構成を分割することはできません。 複製やバックアップを行う場合は、ZFS クローン機能およびスナップショット機能を使用することをお勧めします。 ZFS クローン機能およびスナップショット機能の使用方法については、『ZFS 管理ガイド』を参照してください。 スナップショットを再帰的に送信してシステム全体の複製プロセスを向上させる方法については、RFE 6421958 を参照してください。
    ZFS クローン機能およびスナップショット機能のほかに、Sun StorageTek Availability Suite 製品では ZFS ファイルシステムのリモート複製機能も用意されています。 AVS/ZFS のデモはここでご覧になれます。
    複製またはバックアップを行う目的で、ミラー化された ZFS 構成を分割しようとする場合は、次の点に注意してください。
    1* ミラー化された ZFS 構成の分割は、ZFS ではサポートされていません。 RFE 6421958 に、この機能についての説明があります。
    1* ミラー化された ZFS 構成からディスクを削除し、ディスク上のデータをバックアップしてから、このデータを使用して複製プールを作成することはできません。
    1* ZFS スナップショット機能の代わりにハードウェアレベルのバックアップまたはスナップショット機能を使用する場合は、次の手順を実行する必要があります。
      1. プールをエクスポートする (zpool export プール名)
      2. ハードウェアレベルのスナップショットを実行する
      3. プールをインポートする (zpool import プール名)
        1* ディスクを削除したり、稼動中のプールの一部であるハードウェアを変更したりすることによって、ミラー化された ZFS ストレージプールを分割しようとすると、データが破損する可能性があります。

ZFS と他の製品との連携に関する質問

  1. ZFS はクラスタ環境でサポートされていますか
  2. ZFS をサポートしている他社製のバックアップ製品はありますか
  3. ZFS は SAN 接続デバイスで動作しますか
  1. ZFS はクラスタ環境でサポートされていますか
    SunCluster 3.2 では、Solaris 10 11/06 リリースに対して可用性の高い (HA) ローカル ZFS ファイルシステムがサポートされます。 このサポートでは、システム間でプールを自動的にインポートすることによる、システム間での稼働中のフェイルオーバーが可能です。
    SunCluster 3.2 を使用して可用性の高いローカル ZFS ファイルシステムを構成する場合は、次の点に注意してください。
    ZFS ストレージプールに構成済み定足数デバイスを追加しないでください。 構成済みの定足数デバイスをストレージプールに追加すると、ディスクが再ラベル付けされて定足数構成情報が失われます。 つまり、ディスクがクラスタに定足数投票を提供しなくなるということです。 ディスクを定足数デバイスとして構成できるのは、ストレージプールにディスクを追加したあとです。 あるいは、ディスクを構成解除し、ストレージプールに追加したあとで、定足数デバイスとして再構成することはできます。
    Solaris Cluster 3.2 は、OpenSolaris リリースまたは Nevada リリースではサポートされていません。 オープンソースの Solaris Cluster バージョンの使用方法については、高可用性クラスタに関する OpenSolaris コミュニティーのページを参照してください。
    ZFS はネイティブクラスタ、分散型、並列ファイルシステムのいずれでもないため、複数の異なるホストからの並行アクセスには対応していません。 ZFS は分散型 NFS 環境で共有するときに高い機能を発揮します。
    長期的には、Sun では並行アクセスを可能にするために ZFS をネイティブクラスタファイルシステムとして開発していく計画です。 この作業の具体的スケジュールは未定です。
  2. ZFS をサポートしている他社製のバックアップ製品はありますか
    1* EMC Networker 7.3.2. は ZFS ACL を含む ZFS ファイルシステムをバックアップおよび復元します。
    1* Veritas Netbackup 6.5 は ZFS ACL を含む ZFS ファイルシステムをバックアップおよび復元します。
    1* IBM Tivoli Storage Manager クライアントソフトウェア (5.4.1.2) は、CLI と GUI の両方で ZFS ファイルシステムをバックアップおよび復元します。 ZFS ACL も保持されます。
    1* Computer Associates の BrightStor ARCserve 製品は ZFS ファイルシステムをバックアップおよび復元しますが、ZFS ACL は保持されません。
  3. ZFS は SAN 接続デバイスで動作しますか
    はい、ZFS は直接接続したデバイスおよび SAN に接続したデバイスで使用できます。 ただし、ストレージプールにミラーデバイスまたは RAID-Z の最上位デバイスが含まれていない場合は、ZFS はチェックサムエラーを報告できるだけで、修正まではできません。 ストレージプールに SAN 接続デバイス内のストレージを使用して構築したミラーデバイスまたは RAID-Z デバイスが含まれている場合は、ZFS はチェックサムエラーを報告して修正できます。
    たとえば、内部的にミラー化されたディスクに基づく SAN ファブリックに LUN を提供するように設定された、SAN 接続ハードウェア RAID アレイについて考えてみます。 このアレイから単一の LUN を使用してシングルディスクプールを構築した場合、プールには、検出されたエラーを修正するために ZFS で必要となる複製データは含まれません。 この場合、ZFS はアレイによって発生したエラーを修正できません。
    このアレイから 2 つの LUN を使用してミラー化ストレージプールを構築したり、3 つの LUN を使用して RAID-Z ストレージプールを作成したりする場合は、ZFS には、検出されたエラーの修正に使用可能な複製データが含まれます。 この場合、ZFS はアレイによって発生したエラーの大部分を修正することができます。
    ZFS ストレージプールにミラーデバイスや RAID-Z 最上位仮想デバイスがない場合には常に、プールの存続可能性は背後のストレージデバイスの信頼性に完全に依存します。
    ZFS ストレージプールに SAN 接続ストレージか直接接続ストレージかを問わず単一のデバイスしか含まれない場合は、RAID-Z、動的ストライプ化、I/O 負荷分散などの機能を利用することはできません。
    知らないうちに破損したデータは必ず ZFS で検出されます。 一部のストレージアレイでは、チェックサムエラーを検出できても、次の種類のエラーを検出できない場合があります。
    1* 予期しない上書きや疑似書き込み
    1* 宛先を誤った読み取りや書き込み
    1* データパスエラー
    全般的に見れば、ZFS は SAN 接続デバイスに対して設計どおりに動作しますが、SAN 接続デバイスよりシンプルなデバイスで ZFS を使用する場合は、使用可能なすべての機能をさらに活用できます。
    つまり、SAN 接続デバイスで ZFS を使用する場合に ZFS の自己修復機能を利用するには、ZFS ストレージプールで冗長性を構成する必要があります。これは、より低いハードウェアレベルで冗長性が使用可能な場合にも当てはまります。
