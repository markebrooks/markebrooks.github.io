ZFS 产品发行问题

  1. 如何获得 ZFS？
  2. ZFS 何时可用于 <需在此处插入 OS>？
  3. ZFS 表示什么？
  1. 如何获得 ZFS？
    下列发行版中提供了 ZFS：
    1* Solaris Nevada 发行版，内部版本 27a 及更高版本
    1* Solaris Express 发行版
    1* 从 Solaris 10 6/06 发行版开始
  2. ZFS 何时可用于 <需在此处插入 OS>？
    现在有几个项目正在进行中，这些项目要将 ZFS 移植到 FreeBSD 和 Linux（使用 FUSE）。有关 CDDL 的更多信息，请参见有关许可的常见问题解答。
  3. ZFS 表示什么？
    最初，ZFS 是“Zettabyte File System（Zettabyte 文件系统）”的首字母缩写。我们希望实现的最大 SI 前缀是 "zetta"（"yotta" 是不可能实现的）。由于 ZFS 是一个 128 位文件系统，因此该名称表示 ZFS 可以存储 256 x 1015 ZB（其中每 ZB 是 270 字节）。随着时间的推移，除了 128 位容量之外，ZFS 还具备很多其他功能，如稳定的数据完整性、易于管理以及简化的数据管理模型。

ZFS 技术问题

  1. 为什么 ZFS 具有 128 位容量？
  2. ZFS 有哪些限制？
  1. 为什么 ZFS 具有 128 位容量？
     实践表明，文件系统的生存期比大多数传统软件的生存期长得多，这在一定程度上是由于盘上格式很难更改的缘故。考虑到 UFS 的当前格式（大部分）已沿用了近 20 年，因此让 ZFS 至少在将来 30 年内保持格式不变也是合理的期望。关于这一点，摩尔定律开启了存储领域的新纪元，由此我们可以预测，将能够在单个文件系统中存储 64 位以上的数据。有关此主题的更详细说明，以及 128 位足以满足需要的原因，请参见 Jeff 的博客文章。
  2. ZFS 有哪些限制？
    ZFS 的存储限制上限值非常之大，以至于在任何实际的操作中不会遇到任何存储限制。ZFS 可以在每个存储池、文件系统、文件或文件属性中存储 16 x 1018 字节。ZFS 可以存储几十亿个名称：目录中的文件或目录、文件系统中的文件系统，或文件系统的快照。ZFS 可以存储几万亿个项目：文件系统中的文件、文件系统、卷，或池中的快照。

ZFS/UFS 比较问题

  1. 为什么 ZFS 没有类似于 fsck 的实用程序？
  2. 为什么 du(1) 为 ZFS 和 UFS 报告的文件大小不同？
  3. 为什么 ZFS 没有用户或组配额？
  1. 为什么 ZFS 没有类似于 fsck 的实用程序？
     具有类似于 fsck 的实用程序的基本原因有两个：
    1* 检验文件系统完整性－很多时候，管理员只想要确保其文件系统中没有出现盘上损坏。对于大多数文件系统而言，当文件系统处于脱机状态时，就需要运行 fsck。这可能既耗时，成本又高。而 ZFS 能够在系统运行时“清理”池中的所有数据，从而可以在该过程中查找并修复任何错误数据。根据将来的计划，会增强此功能以便实现后台清理。
    1* 修复盘上状态－如果计算机崩溃，某些文件系统的盘上状态将不一致。添加的日志记录功能已经解决了其中的部分问题，但无法滚动日志仍可能导致文件系统需要修复。这种情况下，就会出现为人所熟知的异常错误（如在更新父链接前创建目录条目），不过这些错误能够可靠地修复。ZFS 不会遇到此问题，因为盘上数据始终是一致的。
    如果硬件或软件出现故障，则会引发更具危害性的问题。即使某些文件系统或卷管理器具有基于块的校验和功能，也易于出现各种其他异常，进而生成有效但损坏的数据。这种情况下，故障模式本质上是随机的，大多数文件系统将出现紧急情况（如果是元数据），或无提示地将错误数据返回到应用程序。在上述任一情况下，fsck 实用程序几乎没有作用。由于这种损坏与已知异常不符，因此很可能无法修复它。如果使用 ZFS，从统计学意义上讲，上述错误在冗余配置中是不存在的。在非冗余配置中，会正确地检测到这些错误，但在尝试读取块时会导致 I/O 错误。理论上可以编写一个工具来修复此类损坏，但进行这样的尝试可能要使用特殊的一次性工具。当然，ZFS 同样易于受软件错误的影响，但这些错误所导致的一致性损坏模式使用普通工具即可修复。在 5 年的 ZFS 开发过程中，还没有出现过这样的模式。
  2. 为什么 du(1) 为 ZFS 和 UFS 报告的文件大小不同？
    在 UFS 中，du(1) 会报告文件中数据块的大小。在 ZFS 中，du(1) 会报告磁盘上所存储的文件的实际大小，其中包括元数据和压缩数据。此报告确实有助于回答以下问题：“如果删除此文件，将释放出多少额外空间？”。因此，即使在压缩处于关闭状态时，您仍会看到 ZFS 和 UFS 之间的不同结果。
  3. 为什么 ZFS 没有用户或组配额？
    ZFS 文件系统可用作逻辑管理控制点，这样，您便可以查看使用情况、管理属性、执行备份、实施快照等。对于起始目录服务器，ZFS 模型使您可以轻松地基于每个用户设置一个文件系统。由于文件系统是管理控制点，因此 ZFS 配额并不特意与某个特定用户关联。
    可以在能够代表用户、项目、组等的文件系统上设置 ZFS 配额，也可以针对文件系统分层结构的整个部分设置 ZFS 配额。这样，便可以通过某些方式合并配额，而这些方式是传统按用户配额所无法实施的。之所以引入按用户配额，是因为多个用户必须共享同一文件系统。
    ZFS 文件系统的配额比较灵活，而且易于设置。可以在创建文件系统时应用配额。例如：
# zfs create -o quota=20g tank/home/users

在此文件系统中创建的用户文件系统会自动继承在父文件系统上设置的 20 GB 配额。例如： |

# zfs create tank/home/users/user1
# zfs create tank/home/users/user2
# zfs list -r tank/home/users
NAME                    USED  AVAIL  REFER  MOUNTPOINT
tank/home/users        76.5K  20.0G  27.5K  /tank/home/users
tank/home/users/user1  24.5K  20.0G  24.5K  /tank/home/users/user1
tank/home/users/user2  24.5K  20.0G  24.5K  /tank/home/users/user2

如果增大 ZFS 存储池中的磁盘空间，则可以在文件系统处于活动状态时增大 ZFS 配额，而不需要任何停机时间。
ZFS 小组正致力于改进多文件系统的管理，而不是尝试使基于用户的配额适合基于文件系统（以文件系统作为控制点）的管理模型。
对于用来包含邮件所用磁盘空间的基于用户的配额，另一种替代方法是使用具备配额功能的邮件服务器软件，如 Sun Java System Messaging Server。此软件提供了用户邮件配额、配额警告消息、配额到期和清除功能。

ZFS 管理问题

  1. 如果每次引导时 ZFS 都出现紧急情况，应该怎么做？
  2. ZFS 是否支持热备件？
  3. 能否从 ZFS 池中移除设备？
  4. 能否将 ZFS 用作根文件系统？那么能否用于区域 (zone) 呢？
  5. 能否分割镜像的 ZFS 配置？
  1. 如果每次引导时 ZFS 都出现紧急情况，应该怎么做？
    在设计上，ZFS 通过采用冗余技术（镜像或 RAID-Z）使之在发生任何硬件故障时均可保持完好。遗憾的是，非复制配置中的某些故障可能导致在尝试装入池时 ZFS 出现紧急情况。这个错误会在不久的将来得到修复（还会增强另外几个比较有用的功能，如后台清理）。同时，如果发现您自己遇到因池损坏而无法引导的情况，请执行以下命令：
    1. 使用 '-m milestone=none' 引导
    2. # mount -o remount /
    3. # rm /etc/zfs/zpool.cache
    4. # reboot
       这将从系统中删除有关池的所有信息。您将必须重新创建池并从备份恢复。
  2. ZFS 是否支持热备件？
    支持，ZFS 热备件功能可用于 Solaris Express Community Release（内部版本 42）、Solaris Express July 2006 发行版和 Solaris 10 11/06 发行版。有关热备件的更多信息，请参见 ZFS 管理指南。
  3. 能否从 ZFS 池中移除设备？
    当前不支持移除顶级 vdev，如整个 RAID-Z 组或未镜像配置中的磁盘。计划在将来的发行版中提供此功能。
    可以使用 zpool detach 命令从镜像的 ZFS 配置中移除设备。
    可以使用 zpool replace 命令在镜像或 RAID-Z 配置中将设备替换为相同大小的设备。
  4. 能否将 ZFS 用作根文件系统？那么能否用于区域 (zone) 呢？
    ZFS 文件系统当前在 Solaris 10 发行版上不能用作根文件系统。但在适用于 x86 系统的 SX 社区发行版中提供了 ZFS 根和引导支持的一个小子集。有关更多信息，请参见 ZFS Boot。敬请等待 Solaris 10 ZFS Boot 发行计划的消息。
    在 Solaris Express 发行版中，ZFS 可以用作区域根路径，但在相关工具识别 ZFS 文件系统之前不能对其进行修补或升级。有关更多信息，请参见有关区域的常见问题解答。
    此外，不能在 ZFS 文件系统上创建 cachefs 高速缓存。
  5. 能否分割镜像的 ZFS 配置？
    当前，ZFS 无法分割镜像配置以进行克隆或备份。进行克隆和备份的最佳方法是使用 ZFS 克隆和快照功能。有关使用 ZFS 克隆和快照功能的信息，请参见 ZFS 管理指南。请参见 RFE 6421958，以便通过递归方式发送快照，从而改进跨系统的复制过程。
    除了 ZFS 克隆和快照功能以外，Sun StorageTek Availability Suite 产品还提供了 ZFS 文件系统远程复制功能。此处提供了 AVS/ZFS 演示。
    如果尝试分割镜像 ZFS 配置以进行克隆或备份，请记住以下注意事项：
    1* ZFS 不支持分割镜像的 ZFS 配置。编写了 RFE 6421958 以便提供此功能。
    1* 无法从镜像 ZFS 配置中移除磁盘、备份磁盘上的数据，然后使用此数据创建克隆池。
    1* 如果要使用硬件级备份或快照功能而不是 ZFS 快照功能，则需要执行下列步骤：
      1. zpool export pool-name
      2. 硬件级快照步骤
      3. zpool import pool-name
        1* 通过移除磁盘或更改活动池中包含的硬件来分割镜像 ZFS 存储池的任何尝试都可能会导致数据损坏。

ZFS 与其他产品交互问题

  1. 群集环境中是否支持 ZFS？
  2. 哪些第三方备份产品支持 ZFS？
  3. ZFS 能否与 SAN 连接的设备一起使用？
  1. 群集环境中是否支持 ZFS？
    Solaris Cluster 3.2 支持本地 ZFS 文件系统在 Solaris 10 11/06 发行版中用作高可用性 (highly available, HA) 系统。此支持允许在系统间进行实时故障转移，以及在系统间自动导入池。
    如果使用 Solaris Cluster 3.2 将本地 ZFS 文件系统配置为高可用性系统，请留意以下注意事项：
    不要将已配置的法定设备添加到 ZFS 存储池中。如果将已配置的法定设备添加到存储池中，将重新标记磁盘，并因此丢失法定配置信息。这意味着磁盘不再向群集提供法定选票。将磁盘添加到存储池之后，您可以将该磁盘配置为法定设备。或者，您也可以先取消配置磁盘，并将它添加到存储池中，然后将该磁盘重新配置为法定设备。
    OpenSolaris 或 Nevada 发行版不支持 Solaris Cluster 3.2。有关使用开放源代码 Solaris Cluster 版本的信息，请转至开放式高可用性群集社区页面。
    ZFS 不是本地群集，也不是分布式文件系统或并行文件系统，因此不能实现多台不同主机的并发访问。在分布式 NFS 环境中共享时，ZFS 表现优异。
    从长远观点出发，我们计划实现将 ZFS 作为本地群集文件系统，以便实现并发访问。目前还未开展这项工作。
  2. 哪些第三方备份产品支持 ZFS？
    1* EMC Networker 7.3.2 可备份和恢复 ZFS 文件系统，包括 ZFS ACL。
    1* Veritas Netbackup 6.5 可备份和恢复 ZFS 文件系统，包括 ZFS ACL。
    1* IBM Tivoli Storage Manager 客户机软件 (5.4.1.2) 可使用 CLI 和 GUI 备份和恢复 ZFS 文件系统。同时还将保留 ZFS ACL。
    1* Computer Associates 的 BrightStor ARCserve 产品可备份和恢复 ZFS 文件系统，但不保留 ZFS ACL。
  3. ZFS 能否与 SAN 连接的设备一起使用？
    可以，ZFS 可以与直接连接的设备或 SAN 连接的设备一起使用。但是，如果存储池不包含镜像或 RAID-Z 顶级设备，则 ZFS 只能报告校验和错误，但不能更正这些错误。如果存储池由镜像或 RAID-Z 设备（使用 SAN 连接的设备中的存储构建）组成，则 ZFS 可以报告并更正校验和错误。
    例如，考虑一个 SAN 连接的硬件 RAID 阵列，设置该阵列以便向基于其内部镜像磁盘的 SAN Fabric（结构）提供 LUN。如果使用此阵列中的一个 LUN 来构建一个单磁盘池，则该池将不包含 ZFS 更正检测到的错误所需的副本数据。这种情况下，ZFS 无法更正该阵列引入的错误。
    如果使用此阵列中的两个 LUN 来构建镜像存储池，或使用三个 LUN 来创建 RAID-Z 存储池，则 ZFS 将拥有可用于更正检测到的错误的副本数据。这种情况下，ZFS 通常可更正该阵列引入的错误。
    在 ZFS 存储池缺少镜像或 RAID-Z 顶级虚拟设备的所有情况下，存储池的生存能力完全取决于底层存储设备的可靠性。
    如果 ZFS 存储池只包含一个设备，无论该设备是来自 SAN 连接的存储还是直接连接的存储，您都无法利用 RAID-Z、动态条带化、I/O 负载平衡等功能。
    ZFS 始终可以检测到无提示数据损坏。某些存储阵列可检测到校验和错误，但可能无法检测到以下种类的错误：
    1* 意外覆写或虚写
    1* 错误定向读写
    1* 数据路径错误
    总的来说，ZFS 的设计考虑到与 SAN 连接的设备协同使用，但如果为 ZFS 呈现更简单的设备，则可以更好地利用所有可用功能。
    总之，如果将 ZFS 与 SAN 连接的设备一起使用，可通过在 ZFS 存储池中配置冗余（即使冗余在较低硬件级别上可用），来充分利用 ZFS 的自我修复功能。
