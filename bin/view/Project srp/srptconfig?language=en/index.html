COMSTAR SRP Target Installation and Configuration

 The SRP Target configuration and installation process is now described in the COMSTAR Admin Guide. The following information is supplemental to that in the Admin Guide.

Contents of the Packages

 The SRP Target consists of three base packages:

  • SUNWsrptr -- the srpt device driver and associated SMF service.
  • SUNWibdma -- the Infiniband Device Management Agent or (ibdma) device driver. IBDMA advertises target-side services such as the SRP target, so that initiator systems can locate and connect to them.
  • SUNWsrptu -- some MDB walkers for srpt data structures (optional).

 You will need to install these packages from your SXCE or OpenSolaris distribution.

Configuring the SRP Target Service

 To enable the SRP Target, enable the corresponding SMF service: svc:/system/ibsrp/target:default.

svcadm enable ibsrp/target

 The SRP Target service is dependent on the STMF service. To enable SRP along with all of its dependent services, use

svcadm enable -r ibsrp/target

The STMF view of an SRP SCSI Target

 When the ibsrp/target service is enabled, it enumerates each IB Host Channel Adapter (HCA) in the system and registers each of them as an SRP SCSI target. The SCSI Target is a virtual object that contains a task router and acts as a the connection between the service delivery system (SRP) and the SCSI back end. For the SRP Target, a SCSI target is created for each Infiniband HCA in the system. Each IB HCA will be reflected to STMF as a COMSTAR target named 'eui.'. For example, if there is an IB HCA with a HCA GUID of 0003BA0001002E49, then an STMF SCSI target will be created with the name 'eui.0003BA0001002E49'. This SCSI Target is a portal through which initiator systems can connect to the SCSI resources on the target system (Logical Units or LUs).

 Similarly, incoming connections from each initiator system will be labelled with the 'eui.'. The choice of GUID is up to the initiator system. Some initiator varieties use the HCA GUID to label their outgoing connections; others use the port GUID. But whatever GUID the initiator system includes in the request, that is the GUID that will appear to STMF as 'eui.GUID'. A brief comparison to both Fibre Channel and to iSCSI might be helpful, for those who are more familiar with those kinds of COMSTAR port providers. For FC, initiators and targets are named by the world-wide identifiers (WWNs) of the initiator and target FC ports. COMSTAR establishes one SCSI target for each physical Fibre Channel port, named by the WWN of that port. Each incoming connection is labelled with the WWN of the initiator port that initiated the connection. A FC-based SCSI session links together the WWN of an initiator and the WWN of a target, and a particular target-side Logical Unit with a particular Logical Unit Number. LUN Mapping is the process of assigning which Logical Units can be viewed by which initiators through which targets, and what Logical Unit numbers will be used for each one. In COMSTAR, this accomplished by defining a View for each logical unit.

 For COMSTAR iSCSI, the target is a logical unit that can be created arbitrarily. Each iSCSI target is named by an IQN, which is an arbitrary name assigned when the target is created. A system might have one iSCSI target for the whole system, it might have a separate iSCSI target for each LU, or anywhere in between. The itadm command is used to create the iSCSI targets, to bind them to particular IP interfaces or groups of interfaces, and to control the parameter choices of each iSCSI target. Similarly, each initiator is named by an arbitrary IQN, and this IQN is what STMF uses to refer to each initiator system.

 In SRP as in FC, there is no command to create explicit SCSI targets. (There is no srptadm command.) Instead, SCSI targets are created automatically to reflect each separate storage transport unit that is available. But unlike in FC, a SCSI target is created for the HCA as a whole rather than for each physical IB port on the HCA individually.

The Infiniband View of the SRP Target

 The Infiniband view of target-side services consists of IB ports, Target Channel Adapters, IO Units, IO Controllers, and Service Entries. See for example Figure B.3 in Annex B of the SRP Specification for an SRP-specific picture of this architecture. For the COMSTAR SRP target, we create a virtual set of the target-side resources that fit this model. Starting from the network side, each HCA corresponds naturally to a TCA. And an IB port is a port is a port. Similarly, starting from the other end of this model, SRP specifies that each SCSI Target should be represented as a distinct Service Entry for each IO controller that grants access to that SCSI Target.

 We choose an extremely simple model in which each HCA is represented as a single I/O Unit holding a single SRP-capable I/O Controller advertising a single SCSI Target as a Service Entry. This simple mapping is familiar to other users of SRP, and it allows the single well-known GUID of each HCA to double as the identifier for each other needed target-side entity. This model works the way users of IB services expect it to.

Lets work this through an example. A test system has a single HCA with two physical IB ports, both of which are connected to the IB fabric. The HCA GUID is 3BA0001002E48, the port 1 GUID is 3BA0001002E49, and the port 2 GUID is 3BA0001002E4A. What the initiator system sees directly is the port 3BA0001002E49. When IBDMA is running on the target, the initiator system can deliver a Management Datagram (MAD) to IBDMA listening on that IB port to query what IO Units and IO Controllers and Services are available through that port. The target system will reply that there is a single I/O Unit available through port 3BA0001002E49 If SRP is enabled on the target system, then there will be an SRP-flavor I/O Controller available through that I/O Unit, and the I/O Controller will also have GUID 3BA0001002E48. Finally, there will be a single Service Entry that will advertise the availability of an SRP-based SCSI Target through that I/O Controller. The GUID of the SCSI Target will also be (you guessed it) 3BA0001002E48.

When the initiator system wishes to make a connection to that SCSI target, it will send an SRP-login request via port 3BA0001002E49. The login request will carry the IO Controller's GUID (3BA0001002E48), and also an indication of the specific SCSI Target that is requested (which is also 3BA0001002E48).

A future enhancement to the SRP Target could introduce a command to control the target configuration. This could allow us to define arbitrary SRP-based SCSI targets similar to the iSCSI targets created by itadm. Similarly, the command could create arbitrary SRP-based I/O Controllers, and flexibly define the mapping from physical IB port to which emulated I/O controllers are visible through that port. This would be somewhat similar to iSCSI target portal groups.

Basic COMSTAR Quick-Start Guide for SRP

  • Examine the IB Configuration to make sure your IB devices are visible as expected:
-bash-3.2# cfgadm -lv
Ap_Id                          Receptacle   Occupant     Condition  Information
When         Type         Busy     Phys_Id
hca:3BA0001002E48              connected    configured   ok         VID: 0x15b3, PID: 0x6278,\
#ports: 0x2, port1 GUID: 0x3BA0001002E49, port2 GUID: 0x3BA0001002E4A
unavailable  IB-HCA       n        /devices/ib:3BA0001002E48
ib                             connected    configured   ok         InfiniBand Fabric
unavailable  IB-Fabric    n        /devices/ib:fabric
[...snip...]
  • Enable COMSTAR:
svcadm enable stmf
  • Enable SRP Target:
svcadm enable ibsrp/target
  • Your SRP-based SCSI Target should now be available to STMF:
-bash-3.2# stmfadm list-target -v
Target: eui.0003BA0001002E48
    Operational Status: Online
    Provider Name     : srpt
    Alias             : -
    Sessions          : 0
  • Define at least one COMSTAR logical unit (LU) backing store using sbdadm:
-bash-3.2# mkdir /storage
-bash-3.2# touch /storage/file
-bash-3.2# sbdadm create-lu -s 2m /storage/file
Created the following LU:
              GUID                    DATA SIZE           SOURCE
~--------------------------------  ~-------------------  ~----------------
600144f0da29a32c000049b1e4cc0001      2097152          /storage/file
  • Define at least one View to make the LU visible to the target and initiator you are dealing with. By creating a " default view", we make the LU available to all combinations of SCSI initiator and SCSI target.
-bash-3.2# stmfadm add-view 600144f0da29a32c000049b1e4cc0001
-bash-3.2# stmfadm list-view -l 600144f0da29a32c000049b1e4cc0001
View Entry: 0
    Host group   : All
    Target group : All
    LUN          : 24
  • Now your initiator should be able to view the storage, connect to the target, and see the LU. See below for a brief how-to of how to do this using the OFED initiator on Linux and on VMWare ESX. Once the initiator has connected, the target can see the incoming connection:
-bash-3.2# stmfadm list-target -v
Target: eui.0003BA0001002E48
    Operational Status: Online
    Provider Name     : srpt
    Alias             : -
    Sessions          : 1
        Initiator: eui.0003BA0001002EA5
            Alias: 492e000100ba0300:0003ba0001002ea5
            Logged in since: Fri Mar  6 21:18:37 2009

Brief Initiator How-To

Linux with OFED SRP Initiator

  • Download the Linux OFED packages from here.
  • Run install.pl script with "--all" option. The "all" option will install SRP and tools.
  • If the above fails with package dependency, please install all the packages it asks for. gcc must be installed.
  • Once the install is done, read docs/srp*.
  • To display general characteristics of each initiator-side IB HCA and Port:
ibstat
  • To display all the available SRP target IO Controllers on the network, use one of the following:
srp_daemon -a -o -v
ibsrpdm
  • To add SCSI devices corresponding to a particular target.
  • Connect to appropriate device directory
cd /sys/class/infiniband_srp/srp-mthca0-1
  • Enumerate the remote IO Controllers in a format that add_target will expect. One of the following:
srp_daemon -o -c  (Use -n if you want to set the initiator-extension automatically)
ibsrpdm -c
  • Echo the appropriate line of output onto the system file add_target. For example:
echo id_ext=0003ba0001002eac,ioc_guid=0003ba0001002eac,\
dgid=fe800000000000000003ba0001002ead,\
pkey=ffff,service_id=0003ba0001002eac > add_target
  • Use contents of /var/log/messages to determine which scsi device corresponds to each added target
  • On some Linux, lsscsi should show the newly created device. Using the -H and -v options gives more device information. Alternatively cat /proc/scsi/scsi.
  • Now you can do something with the device, such as mkfs /dev/sdd.

VMware ESX 3.5 Update 3 with Mellanox OFED driver

 We have tested with Mellanox OFED V1.2.0 for VMware ESX 3.5 U3 (Build: 123630)

  1. Download and install VMWare ESX 3.5 and provision appropriate licenses. You can download an evaluation copy of VMWare Infrastructure 3 from the VMWare download site.
  2. Download the InfiniBand OFED Drivers for VMware ESX 3.5 from the Mellanox site
    1* Click products tab ->InfiniBand SW/Drivers -> VMWare Drivers, scroll down, then click download tab
    1* Choose the right driver to download (in this case Download Drivers v1.2.0 for ESX 3.5 U3) to save the installation file to your disk.
    1* Follow the installation guide, which can be located in the Related Documents panel on the upper right corner of the Download page, to install. Section 4.1 has complete install instructions.
  3. You can optionally download from vmware.com the VMWare Infrastructure Client and install it on a Windows box. VMWare VirtualCenter 2.5 Update 3 corresponds to the VMWare ESX 3.5. The VI client is a GUI management application that can be used to identify the storage adapters and scan for LUNs.
