New Connection Management

Connection management issues

To get the basic idea of the issues that the connection management code
is addressing, please read the description of CR [6681185][].

High-level description

To get a feel of the high level design and data abstraction, please read
the header file, ns~_connmgmt.h, which I documented as much as I can:

http://cr.opensolaris.org/the/duckwater_phase0/usr/src/lib/libsldap/common/ns_connmgmt.h.html

The followin is a high level description of the change I sent during one
of our internal review in Jan. of this year:

The basic idea is to separate the code doing connection sharing (Sparks)
and the code not doing it (pre-Sparks). Mixing them as we did turns out
to be not so good an idea. So if possible, I would like to revert most
of the code in ns~_connect.c back to the pre-sparks level, and have a
new file, ns~_connmgmt.c to handle and do all the connection sharing
(multiplexing threads onto the same connection, which is called
conn~_mt in short).

In this design, we will have a structure ns~_conn~_user~_t representing a
user of a connection. And there's a ns~_conn~_mt~_t structure to
represent the ldap connection being shared by multiple ns~_conn~_user~_t.
It contains a linked list of conn_user_t and references the old
Connection structure (which contains the real connection details like
server info, ldap handle, etc). So in a way, we create a new layer on
top of the pre-sparks connection code.

There's also a global ns~_conn~_mgmt~_t which is an abstraction of the
connection pool containing sharable connections. This ns~_conn~_mgmt~_t
has a linked list of the ns~_conn~_mt~_t which in turn has the list of
ns~_conn~_user~_t, representing the current requests being processed by
libsldap.

Most of the new code are in ns~_connmgmt.c, which creates/uses/frees
the ns~_conn~_mgmt~_t, ns~_conn~_mt~_t, and ns~_conn~_user~_t structures
and the interactions between them. These structures are defined in
ns~_connmgmt.h. The changes to other files are designed to be minimum
\- mainly to set up/free the ns~_conn~_user~_t representing the search,
write, getent, and auth (pam~_ldap) operations. In the search state
machine in ns~_reads.c, there's also some code to detect/handle bad
connection and either abort an operation or restart it.

The dependency on ldap~_cachemgr(1M) are specified in ns~_connmgmt.h,
with the data structure, ns~_server~_status~_change~_t defining what
libsldap expects of ldap~_cachemgr(1M). The new door interface are
shown in ns~_cache~_door.h, with ldap_get_chg_cookie_t being a new
element which is used to sync up the configuration between libsldap and
ldap~_cachemgr(1M).

A new dependency is the immediate verification of server status by
ldap\//cachemgr(1M) when libsldap calls \_s_api_removeServer() to
indicate that a server might be down. ldap~_cachemgr(1M) should check
it (via ping or get rootDSE or whatever) right away to get the new
status of the server. And immediately report any status changes to
libsldap if requested.

Also, libsldap expects ldap~_cachemgr(1M) to send the change
notification only when there's real change of status or change of
configuration. Not when the DUAprofile is refreshed. Most of the time,
the DUAprofile would stay the same and server status changes won't
happen much when the profile is refreshed, so libsldap should be
shielded from internal temporary changes in ldap~_cachemgr(1M) when
profile or server list is refreshed.

Michen
