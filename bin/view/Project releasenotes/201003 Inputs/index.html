<Example Entry>Bug ID <num>: GUI Installer Shrinks the Solaris Partition to be Created Even if Enough Available Space is Reported on the Disk Screen

  • Platform information: x86
  • Bug category: Installation
  • Bug Description: When creating a Solaris partition, the GUI installer only utilizes the free space at the end of the disk after the last primary partition. However, the sum of the total space available including gaps between partitions is reported on the disk screen. If the user tries to create a Solaris partition which is larger than the amount of free space available at the end of the disk, the size of Solaris partition is adjusted.
  • Error messages: A warning message is displayed when the Next button is pressed on the disk screen and the Solaris partition is shrunk. The title of the warning message is as follows:
  • Submitter Name: A. User
  • Submitter Email Address: a_user@email.com
Adjustments were made to the new partitions
  • Workaround: Use the Solaris fdisk(1M) tool or the GParted tool to create a Solaris2 partition. These tools enable the utilization of unallocated space between already defined partitions. Select the newly created Solaris2 partition as the target for the installation on the installer disk screen.
  • Cross-references: N/A

Bug ID: 6186 - OpenSolaris package renaming

  • Platform information - Both
  • Bug category - Installation
  • Bug description Over 1600 IPS packages were renamed in this release. The old names for the packages have been retained for now, and will point to the new ones, so operations like package updating should be an automatic process.
  • Error messages: There shouldn't be any error messages. Everything should "just work".
  • Workaround: N/A
  • Cross-references: For a list of the old ==> new package mappings at the time of the renaming, plus their classifications and descriptions, see: https://markebrooks.github.io/bin/view/Project+indiana/Renamed+Packages+in+Build+133 
  • Submitter Name: Vidhya Cidambi for B J Wahl
  • Submitter Email Address: bjw@sun.com

Bug ID 9949: beadm unmount fails when filesystems shared between BEs are also mounted.

  • Platform information - Both
  • Bug category - Installation
  • Bug description - When a BE containing installed zones is mounted the zones datasets are lofs mounted as well. When this BE is unmounted these lofs mounts can cause the ZFS datasets of the BE to fail to unmount.
  • Error messages:
      Unable to unmount newbe.
      unmount failed.
  • Workaround: use beadm umount -f <be_name> to force unmount the BE.
  • Cross-references: N/A
  • Submitter Name: Evan Layton
  • Submitter Email Address: evan.layton@sun.com

Bug ID 5270: Creating multiple pools with the same name confuses ZFS boot process

  • Platform information - Both
  • Bug category - Installation
  • Bug description
    OpenSolaris Installer always creates ZFS root pool with name 'rpool'. This will confuse ZFS boot process if more than one OpenSolaris instances are installed on the same disk. Only Automated Installer is affected, since GUI installer does not support creating multiple OpenSolaris instances on one disk.
  • Workaround
    In case it is desirable to have more OpenSolaris instances installed on one system, make sure each OpenSolaris instance is installed on separate disk.
  • Submitter Name:Jan Damborsky
  • Submitter Email Address: jan.damborsky@sun.com

Bug ID 6558646: DM_CTYPE attribute is set to "unknown" for SATA controllers handled by "sata" driver

  • Platform information - x86
  • Bug category - Installation
  • Bug description
    Automated Installer allows to select target disk for the installation by specifying disk selection criteria in AI manifest. Type of disk controller is one of criteria user can
    define in AI manifest (by means of <target_device_type> disk selection tag). Following values are currently supported (they are case insensitive):
    • SCSI
    • ATA
    • Fibre Channel
    • USB

Information about controller type is currently not available for SATA drives with device name in c#t#d# format (handled by Solaris SATA framework).

Information about disk controller type can be obtained on AI client by running Target Discovery test driver when Automated Installer is booted (it has to be run with root privileges):

# /opt/install-test/bin/test_td -dv
Disk discovery
Total number of disks: 1
-------------------------------------------------------------------------------
 num |    name|    vendor|  ctype| mtype| rem| lbl| bsize|#of blocks|size [MB]|
-------------------------------------------------------------------------------
   1 |*   c7d0|   unknown|    ata| FIXED|  No|  VF|   512|1953520128|   953867|
-------------------------------------------------------------------------------

Controller type is reported in 'ctype' column.

  • Workaround
    Use other disk selection criteria to select desired SATA disk
  • Related documentation
    • sata(7D) man page
  • Submitter Name:Jan Damborsky
  • Submitter Email Address: jan.damborsky@sun.com

Bug ID 8575: Automated Installation Fails if Slice 0 is Configured on the Client With Less Than 8 GB Disk Space

  • Platform information - Both
  • Bug category - Installation
  • Bug description - Automated Installation Fails if Slice 0 is Configured on the Client With Less Than 8 GB Disk Space. The Automated Installation fails because it runs out of space in slice 0 of the target device. 
  • Error message/s, if any - The following error message is displayed: Auto install failed
  • Workaround - Choose one of the following workarounds:

    1) Create slice 0 of the target device with more than 8 GB space.
    2) If there is another slice with more than 8 GB space, change the Automated Installation manifest to use that slice. For example, if you want to use Slice 4 of your target device c0t0d0, add the following lines to your Automated Installation manifest:
<ai_target_device>
   <target_device_name>c0t0d0</target_device_name>
   <target_device_install_slice_number>4</target_device_install_slice_number>
</ai_target_device>

  • Cross-references to related bugs, issues, and reading material, if any - NA
  • Submitter Name:Vidhya Cidambi for Ethan Quach 
  • Submitter Email Address: ethan.quach@sun.com 

Bug ID 8661: Automated Installer Fails to Continue With Installation if Slices from the Previous Configuration Exist

  • Platform information - Both
  • Bug category - Installation
  • Bug description - When a custom Automated Installation manifest is used to install a system and there are customizations in the manifest to create slices other than the root slice if those slices already exist on the disk in your system, then the automated installation fails. 
  • Error message/s, if any - The following error message is seen in the install_log file:

  root@opensolaris:/tmp# tail -20 /tmp/install_log

  /tmp/ai_combined_manifest.xml is a valid manifest
  Auto reboot disabled
  Disk = c8t0d0 found on the system
  Disk name selected for installation is c8t0d0
  no manifest partition information found
  disk partition info not changed
  System reports enough physical memory for installation,
  swap is optional
  creating slice which already exists
  failed to modify slice(s) specified in the manifest
  Auto install failed 

  • Workaround - Use the format(1M) command to remove the existing slices from your disk. Rerun the automated installation by rebooting your system back into the Automated Installer.
  • Cross-references to related bugs, issues, and reading material, if any
  • Submitter Name:Vidhya Cidambi for Ethan Quach
  • Submitter Email Address:ethan.quach@sun.com

Bug ID 5003: Default dumpadm Configuration Is Not Set to Capture Crash Dumps

  • Platform information - Both
  • Bug category - System Administration
  • Bug description - The dumpadm config in the OpenSolaris release is not created by default or set to save crash dumps. 
  • Error message/s, if any - n/a
  • Workaround - Note: This release note is carried forward from the OpenSolaris 2009.06 release.

Use the dumpadm(1M) command to enable crash dumps at boot time:

# dumpadm -y

Alternatively, you can create a cron job that enables the saving of crash dumps at a later time.

  • Cross-references to related bugs, issues, and reading material, if any
  • Submitter Name:Juanita Heieck
  • Submitter Email Address:juanita.heieck@sun.com

Bug ID 14894: beadm activate fails when rpool is on a logical partition

  • Platform information - both
  • Bug category - Install
  • Bug description - Because the -m option to installgrub which over-writes the MBR is required for extended partitions it is possible that beadm activate can fail due to an installgrub failure. This is because we do not try to force the over-writing of the MBR when calling installgrub from within beadm.
  • Error message/s, if any:

From beadm:
Unable to activate newbe.
Unknown external error.

From pkg image-update:
pkg: unable to activate opensolaris-1

  • Workaround:

# beadm create new_be
# beadm mount new_be /a
# pkg -R /a image-update
# cp /a/boot/grub/capability /rpool/boot/grub/
# installgrub -fm /a/boot/grub/stage1 /a/boot/grub/stage2 /dev/rdsk/c?t?d?s?
# beadm umount new_be
# beadm activate new_be
If your root pool is mirrored the installgrub command will need to be run for
all the devices in your pool.

  • Cross-references: N/A
  • Submitter Name: Evan Layton
  • Submitter Email Address: evan.layton@sun.com

Bug ID: 7802 - setup-service lookup fails/delete-service dns-sd killing fails, if service name length is long

  • Platform information: Both
  • Bug category: Installation
  • Bug description: Using an Automated Installer service name longer than 59 characters will cause installadm commands to fail
  • Error behavior: A dns-sd process will be left running on the system after a delete-service, if the name is longer than 59 characters. If a name longer than 64 characters is used, then the create-service command will fail leaving orphaned files untracked by any Automated Installer service.
  • Workaround with explicit directions for one or more workaround options: Do not use Automated Installer service names longer than 59 characters.
  • Cross-references to related bugs, issues, and reading material, if any:
  • Submitter Name: Clay Baenziger
  • Submitter Email Address: clayb@sun.com

Bug ID: 6922 -  installadm create-service should have option to specify ip address passed to dhcp config

  • Platform information: Both
  • Bug category: Installation
  • Bug description: The OpenSolaris Automated Installer does not support multihomed servers
  • Error message/s, if any:
  • Workaround with explicit directions for one or more workaround options:
      At this time it is not recommended for an administrator to use the Automated Installer on a server with multiple network connections. If one must the following data sources must be modified:
  • DHCP
            -Which subnet to configure on
            -Which router to provide
            -Which bootfile location to provide
            (One can by-hand maintain DHCP entries modifying BootSvrA, BootFile macro values as appropritate)
  • mDNS
            -Which IP address to advertise for the AI webserver
            (One can ensure client routing to the IP address advertised by the dns-sd(1M) process running on Automated Installer server)
  • GRUB
            -Which install_media IP to provide
            -Which install_svc_address IP to provide
             (One can hand edit /tftpboot/menu.lst.<service name> to be accurate)
  • Wanboot
            -Which network to put wanboot.conf files under
             (One can be solved by creating symlinks with ln -s <src> <tgt> for all networks served in /etc/netboot)
  • Cross-references to related bugs, issues, and reading material, if any:
    Bug 6182 -  installadm tools doesn't support install servers which have mutiple subnets
    Bug 7115 -  Custom wanboot.conf files are ignored in AI servers with multiple NIC cards
    Bug 7149 -  installadm should allow user to choose which subnets to use
  • Submitter Name:Clay Baenziger
  • Submitter Email Address:clayb@sun.com

Upgrade to 2010.03 with little spare memory fails

  • Platform information - Both
  • Bug category - System Administration
  • Bug description:
        When image-updating on a low-memory machine (less than a gig, maybe more, depending on number of installed packages), you may run out of memory.
  • Error message/s, if any
        Out of memory error
  • Workaround with explicit directions for one or more workaround options
        - Before running image-update, execute "pfexec pkg install SUNWipkg", then execute "pfexec pkg image-update -f".  The -f option will reduce memory needed to perform the image-update by skipping the up-to-date check for the package client.
        - Close as many programs as possible before running the update.
        - Terminate the window system (svcadm disable -t gdm) and do the operation on the console, or logged in remotely
        - Consider starting the system in single-user mode before running the update
        - Finally, try adding more swap or more RAM.
  • Cross-references to related bugs, issues, and reading material, if any
        See bug 9388
  • Submitter Name: Danek Duvall
  • Submitter Email Address: danek.duvall@sun.com

15362: AI manifests are installed world-readable

  • Platform information - Both
  • Bug category - Installation
  • Bug description
        The XML manifest files used by Automated Installation (AI) are readable by any user on the AI server, and openly accessible over the network via the AI http manifest service.  Any passwords provided as part of the configuration manifest should not be considered to be secure.
  • Error message/s, if any
        None
  • Workaround with explicit directions for one or more workaround options
        - To limit readability of the manifests on the AI server, use the command chmod -R og-r /var/ai/*/AI_data
        - Access to manifests over http can be restricted using IPfilter to limit access to the manifest service ports to only specific networks or clients.
        - On first boot after installation of a system using AI, login and change any passwords configured using AI.  For best security, boot the system to single-user to do this.  On SPARC systems, single-user is selected by adding the "-s" option to the boot command.  On x86/x64 systems, single-user is selected by editing the GRUB menu interactively and appending the -s option to the kernel$ command.
  • Cross-references to related bugs, issues, and reading material, if any
  • Submitter Name: Dave Miner
  • Submitter Email Address: dave.miner@oracle.com
